5.1

Broker、主题、发布订阅

学习目标

三角色

发布者、订阅者、Broker

主题

用频道名隔离不同业务

payload

消息正文，如 on / off

本课拓扑

手机下行控灯；板子上行报状态

发布订阅模型

MQTT 不强调「你直接连我」，而强调「大家都连中转站」。Broker（中转站，本课常用软件 Mosquitto）按主题（频道名，如 course/led/cmd）把消息转给订阅了该主题的客户端。

手机 App（发布者） │ publish course/led/cmd payload="on" ▼ Broker（Mosquitto） │ 转发给所有订阅者 ▼ 荔枝派 4A 程序（订阅者）→ GPIO → 外接 LED 反方向： 板子 publish course/led/status payload="on" → Broker → 手机 / 桌面客户端看见状态
术语一句话本课例子
Broker消息中转站主机或板上的 Mosquitto，默认端口 1883
主题频道名course/led/cmd、course/led/status
发布往某主题发一条手机发 on
订阅盯着某主题收板子订 course/led/cmd
payload消息正文on / off / JSON 状态

启动 Mosquitto（主机示例）

# Debian/Ubuntu 示例；以你发行版包名为准
$ sudo apt update
$ sudo apt install -y mosquitto mosquitto-clients
$ sudo systemctl enable --now mosquitto

# 本机自测：一终端订阅，另一终端发布
$ mosquitto_sub -h 127.0.0.1 -t 'course/led/cmd' -v
$ mosquitto_pub -h 127.0.0.1 -t 'course/led/cmd' -m 'on'
网络：板子连接 Broker 时用主机局域网 IP，不要写 127.0.0.1（那是板子自己）。防火墙需放行 1883。课堂若提供统一 Broker 地址，以当堂板书为准。

课堂练习

  1. 若两个客户端都订同一主题，一方发布后，另一方会怎样？
  2. 为何控灯命令与状态上报要用两个不同主题？
本节成果：能画发布—Broker—订阅三角 · Mosquitto 能收发一条测试消息 · 记下 Broker IP 与主题名。
5.2

下行：订阅命令驱动外接 LED

学习目标

连接

板端程序连上 Broker

订阅

订命令主题，回调里处理 payload

GPIO

复用 gpiod 驱动外接 LED

约定

on/off（大小写按脚手架约定）

主题与 payload 约定

本课默认：

  • 命令主题：course/led/cmd
  • payload：on → 灯亮；off → 灯灭
  • 未知 payload → 打 [ERR]，不改变当前电平
mosquitto_pub / 手机 App │ topic=course/led/cmd msg=on ▼ Broker ▼ mqtt-led 订阅回调 │ strcmp(payload, "on") ▼ gpiod_line_set_value(LED, 1)

脚手架 chapters/ch05/code/mqtt-led/ 用 libmosquitto（Mosquitto 的 C 客户端库）提供连接与回调骨架；on_message 里点灯逻辑标为 TODO。交叉编译需链接 -lmosquitto -lgpiod；板端安装对应运行库。

跟做步骤

  1. 1
    改宏
    BROKER_HOST 填主机局域网 IP；LED_LINE 按接线表；主题名与课堂一致。
  2. 2
    交叉编译并部署
    见下方命令；file 确认 RISC-V。
  3. 3
    主机发布
    mosquitto_pub -h <broker> -t course/led/cmd -m on，板端 LED 应亮；再发 off 应灭。
$ cd chapters/ch05/code/mqtt-led
$ make clean && make
$ file mqtt-led
$ scp mqtt-led user@board-ip:~/
$ ssh user@board-ip './mqtt-led'

# 主机另一终端
$ mosquitto_pub -h BROKER_IP -t 'course/led/cmd' -m 'on'
$ mosquitto_pub -h BROKER_IP -t 'course/led/cmd' -m 'off'
权限：GPIO 打不开时按第三章方式授权。MQTT 连不上时先在板端 ping BROKER_IP，再查端口与主题拼写。

课堂练习

  1. 订阅回调在哪个线程上下文执行？若回调里做很重的事，可能有什么风险？（概念即可）
  2. payload 写成 ON（大写）时，你的程序应如何处理？
本节成果：板子能订阅并响应 on/off · 外接 LED 可见动作 · 日志有收到主题与 payload 的记录。
5.3

上行 publish 状态 · 深入理解：tcpdump 抓一条 publish 看分层

学习目标

上行

灯状态变化后 publish 到状态主题

联调

手机/mosquitto_sub 能看见状态

抓包

用 tcpdump 抓住一次发布

深入理解

指出 MQTT 落在应用层，下面是 TCP、IP

发布状态

灯亮灭改变后，板子向 course/led/status 发布当前状态（如 on/off）。订阅该主题的客户端即可确认「命令是否落地」。脚手架中 publish_status() 为桩，实验里接到点灯成功之后调用。

/* 伪代码 */
led_set(1);
publish_status("on");   /* TODO: mosquitto_publish(...) */
# 主机订阅状态
$ mosquitto_sub -h BROKER_IP -t 'course/led/status' -v

深入理解：tcpdump 抓一条 publish

tcpdump 在网卡上抓原始包。MQTT 默认跑在 TCP 1883 端口上；再往下是 IP，再往下是以太网帧。你要亲眼看到：应用里的一次 publish，在网上是「被一层层包起来」发出去的。

应用：mosquitto_publish(topic, payload) │ ▼ MQTT 报文（应用层） ▼ TCP 段（传输层，端口 1883） ▼ IP 包（网络层） ▼ 以太网帧（链路层） │ 网线 / Wi-Fi
  1. 1
    在 Broker 主机或板端开抓包（二选一，以能看到板→Broker 流量为准）： 
    sudo tcpdump -i any -nn -A port 1883 -c 20
  2. 2
    触发一次上行
    让板子 publish 状态（或主机 mosquitto_pub 发一条）。
  3. 3
    在输出里找
    1883 端口、ASCII 区可能露出主题名或 payload 片段；结合 -nn 看到的 IP:端口，对照「MQTT 在 TCP 之上」。
深入理解验收话术：「这次 publish 的内容在 MQTT 应用层；它装在 TCP（1883）里，再装在 IP 里发出去。tcpdump 让我看见端口与载荷痕迹，而不是只相信库函数返回成功。」
抓包需相应权限；只抓本课端口、限制 -c 包数。不要在公共网络长时间无过滤抓包。

课堂练习

  1. 若 tcpdump 看不到 1883 流量，按「进程没发 / 走了别的网卡 / 权限不足」列三条检查。
  2. 加密 MQTT（TLS，常 8883）时，ASCII 还能直接看到 payload 吗？说明原因。
本节成果：状态主题可被订阅端看见 · 保存一段含 1883 的 tcpdump 输出 · 能口述 MQTT/TCP/IP 三层关系。
