3.1

外设与 I/O

I/O：物理世界怎么进程序

外设是 CPU 之外、用来感知或驱动外部世界的硬件模块——传感器、LED、风扇驱动等都算。I/O（Input/Output）是程序与这些硬件打交道的总称：输入把外界变成程序里的数，输出把程序里的决定变成外界动作。

物理世界 电气层 软件层 温度 / 按键 → 电压 / 电平 → 驱动读寄存器或缓冲 LED / 风扇 ← 高/低、开关 ← 程序写「开/关」

同一条 GPIO 脚，配置成输出就能驱动 LED；配置成输入就能读按键。传感器、执行器再复杂，对用户态程序来说，最终仍是「读到一个值」或「写出一个命令」。

寄存器、MMIO 与用户态边界

寄存器是芯片内部一组可读写的控制位，用来配置外设、读状态、写输出。MMIO（Memory-Mapped I/O，内存映射 I/O）把这些寄存器映射到 CPU 可见的地址空间：驱动对某个地址的读写，实际落到外设控制逻辑，而不是普通 RAM。

说法含义本课怎么用
MMIO用「访存」方式访问设备寄存器建立心智；用户态程序不直接写这些地址做业务
设备节点如 /dev/gpiochip0应用通过 open / 库 API 访问
内核驱动独占硬件、做权限与时序管理保证多进程安全、屏蔽芯片细节
应用 C 代码 │ gpiod_* / open / ioctl ▼ 设备节点（/dev/...） │ 系统调用 ▼ 内核驱动 ──► MMIO 寄存器 ──► 物理引脚 / 外设
记住一句：板上有 MMIO，但本课业务程序走内核暴露的接口（如 gpiod）。「深入理解」会用 devmem 只读核对某一位是否翻转，用来对照理解驱动写了寄存器——不拿它当日常控灯手段。
安全边界：GPIO 多为 3.3 V 逻辑。不要把引脚直接接到市电或大电流负载；风扇等执行器必须经继电器或驱动模块，GPIO 只控信号端。
本节成果：能口述物理世界 → 电气 → 软件的路径 · 理解 MMIO 与用户态 API 的分工。
3.2

GPIO：用 gpiod 控灯与继电器

GPIO 与 gpiod

GPIO（General-Purpose Input/Output）是可由软件配置为输入或输出的通用引脚。gpiod / libgpiod 是 Linux 上访问 GPIO 的用户态库，通过字符设备 /dev/gpiochipN 申请引脚、读写电平。交叉编译时链接 -lgpiod；板端需有对应运行库。

应用（C） │ gpiod_chip_open / gpiod_line_request_output │ gpiod_line_set_value ▼ libgpiod ──► /dev/gpiochip0 │ ▼ 内核 GPIO 控制器 ──► 物理引脚 ──► LED / 继电器信号端
#define GPIO_CHIP_PATH  "/dev/gpiochip0"
#define LED_LINE        17   /* 以课程接线表为准 */

灯与继电器

控灯：GPIO 输出高/低 → LED 亮/灭（有的板子低电平点亮，以接线说明为准）。

继电器是用小信号开关大电流的模块：GPIO 只驱动继电器的控制端（几毫安级信号），继电器内部的触点再去接通风扇电机的供电回路。这样 GPIO 脚不会直接扛风扇电流。

对象GPIO 做什么注意
LED输出高/低驱动灯脚位以接线表为准，写进宏
继电器 → 风扇输出高/低给继电器信号端风扇电源按套件说明接；共地必接

3.3

温控闭环

传感 → 决策 → 执行

DHT22 是温湿度探头，用一根数据线接 GPIO，周期上报温度和湿度。它按严格时序一位一位收发数据；时序稍紧或线上干扰时，用户态读取可能偶发失败——属预期：打 [ERR] 并进入下一采样周期，让程序继续跑。

执行侧本课主线是开/关：GPIO 高/低 → 继电器 → 风扇电源通断。两档阈值形成滞回（第二章已定义：中间区间保持上一状态），避免温度在临界附近风扇狂抖。

DHT22 ──► 温度 T │ ▼ 阈值判断 T > T_HIGH → 风扇开（继电器吸合） T < T_LOW → 风扇关（继电器释放） 中间区间 → 保持上一状态 │ ▼ gpiod 写 FAN_LINE ──► 继电器信号端 ──► 风扇
/* 伪代码：完整主循环在 lab 的 temp-fan 里落地 */
if (读温度失败) { 打 [ERR]; 继续下一轮; }
else if (T > T_HIGH) 风扇_开();
else if (T < T_LOW)  风扇_关();

脚手架：chapters/ch03/code/temp-fan/。默认 SIMULATE_SENSOR=1 用假温度演示阈值；接好 DHT22 后改为 0。接线、周期采样与验收见 lab.html。

本节成果：能说清 DHT22 → 阈值 → 继电器 → 风扇的数据流。下一节深入理解用 devmem 只读核对「写灯时寄存器位真的翻了」。
深入理解

gpiod 写灯之后：内核如何翻寄存器位，以及你如何亲眼看见

原因：用户态 API 并不直接碰芯片寄存器

你在 C 里调用 gpiod_line_set_value(line, 1)，看起来只是「把灯打开」。真正改硬件的，是内核里的 GPIO 驱动：它通过系统调用接到你的请求后，往某个 MMIO 地址写一笔，把对应引脚那一位从 0 翻成 1（或反过来）。引脚电平变了，LED 才亮。

回顾几个词，串成一条链：

  • GPIO：通用输入输出引脚，本课用来输出高/低控灯或继电器信号端。
  • 继电器：小信号开关大电流；GPIO 只管信号，风扇电源走继电器触点。
  • MMIO：设备寄存器被映射成「像内存一样」的地址；驱动对地址的写，落到硬件控制逻辑。
  • devmem：按物理地址读/写内存映射区域的小工具。本课只读，用来核对某一位是否翻转，绝不拿它当日常控灯手段。
① 应用：gpiod_line_set_value(..., 1) │ 经 libgpiod → 打开的 /dev/gpiochipN │ 系统调用进入内核 ▼ ② 内核 GPIO 驱动 │ 写 MMIO：把该引脚对应的 bit 置 1（或清 0） ▼ ③ 硬件寄存器（某物理地址上的一个字） │ ▼ ④ 物理引脚电平变高 / 变低 ──► LED 亮 / 灭 旁路核对（只读，不当业务路径）： 点灯前：devmem 读该寄存器 → 记下那一位是 0 点灯后：devmem 再读同一地址 → 那一位变成 1

读图：左边主路径是业务程序该走的路——永远经 gpiod / 内核。右边旁路是课堂验证：你不改驱动源码，也能用只读 devmem 看见「那一位确实翻了」，从而相信主路径终点落在寄存器上。

机制落地：点灯前后各读一次

寄存器地址与「哪一位对应哪根脚」来自芯片手册与板卡文档，本课由当堂下发（具体 0x…… 与 bit 编号以课堂材料为准，勿自行猜测）。你要做的是固定同一地址、同一位，做前后对比。

  1. 1
    先用 gpiod 控灯
    跑 blink，或写一个只置一次高/低的小程序，确认 LED 有可见变化。没有可见变化时，先查接线与 line 编号，不要先怪寄存器。
  2. 2
    拿到课堂给出的地址与 bit
    例如：某 GPIO 数据寄存器物理地址 0xADDRESS，你的 LED 对应第 N 位。把地址和 N 记在实验记录里。
  3. 3
    点灯前只读一次
    sudo devmem 0xADDRESS 32，得到一个 32 位十六进制值。按课堂说明标出第 N 位当前是 0 还是 1。
  4. 4
    点灯后再读同一地址
    保持其他条件不变，只改变 gpiod 写出的电平，再跑一次 devmem。对比：第 N 位应随灯的亮灭翻转；其余位通常不变（若变了，说明读错地址或脚位映射理解有误）。
# 地址与位以课堂下发为准；以下仅为命令形态
$ sudo devmem 0xADDRESS 32
# 例如输出：0x00000000  → 记下第 N 位

# 用 gpiod 把灯打开（blink 或单独 set）

$ sudo devmem 0xADDRESS 32
# 例如输出：0x00020000  → 同一第 N 位已翻
纪律：本课对 MMIO 只做 devmem 读。不要用它写寄存器控灯——会与内核驱动抢硬件，行为不可预期，也可能把脚配成危险状态。

对照：内核驱动里「写寄存器」的那一行

课堂会指出（或投影）内核 GPIO 驱动中与「写数据寄存器」对应的那一行——形态往往是「读出旧值 → 改某一 bit → 写回」或直接写置位/清零寄存器。你不需要在本课改内核；目的是把三件事对齐：

用户态一行： gpiod_line_set_value(line, 1); 内核驱动一行（课堂指出的写寄存器语句）： 写 MMIO → 改变第 N 位; 你亲眼看见： devmem 前后对比 → 第 N 位 0→1（或 1→0） LED 同时亮灭变化

读图：上层 API、驱动写语句、硬件位、肉眼可见的灯，四者应同时一致。若灯亮了但位没变，多半是地址/位号对错了脚；若位变了灯不亮，多半是接线或高低电平有效极性问题——回到 3.2 查硬件，而不是怀疑「Linux 魔法」。

整条故事：gpiod 写灯 → 系统调用 → 内核 GPIO 驱动写 MMIO → 寄存器某一 bit 翻转 → 引脚电平变 → LED 亮；用 devmem 只读在点灯前后核对同一 bit，并对照课堂给出的驱动写寄存器那一行。

本节成果：能口述 GPIO / 继电器 / MMIO / devmem · 能用 blink + devmem 只读完成位翻转对照 · 带着概念进入本章实验组装完整温控闭环。
