1.1

荔枝派 4A 与 RuyiSDK

认识荔枝派 4A

荔枝派 4A 是本课用的 RISC-V 开发板：一块带 CPU、内存、存储接口和排针的电路板，用来跑 Linux、接传感器和执行器。

指令集是 CPU 认识的机器指令清单——规定「加、跳、读内存」等操作怎么编码成二进制。电脑常见 x86_64 指令集；荔枝派 4A 跑的是 riscv64 指令集。两边指令编码不同，所以电脑上用普通 gcc 编好的程序，不能直接丢到板子上跑。

RISC-V 是一种开放的指令集架构：规范公开，任何人可以做兼容芯片。本课板子上的 Linux 系统是 RevyOS——面向 RISC-V 板子的 Linux 发行版。

RISC-V 指令集（规定 CPU 认哪些指令） ↓ 荔枝派 4A（开发板：CPU + 内存 + 接口） ↓ RevyOS（板子上的 Linux 系统） ↓ RuyiSDK / ruyi（电脑上装工具链与镜像的入口）

动手前先在板子上标出：电源口、串口排针、网口、启动介质（eMMC / SD）。串口模块必须是 3.3V 电平——电平是引脚高低电压的约定；5V 可能损坏板子。

项目要求检查方式

常见陷阱
· 部分 USB-C 线只供电、不传数据 → 电脑认不出设备时先换线
· 串口电平必须 3.3V → 5V 可能损坏板卡
· 工具链前缀选 linux-gnu（跑 Linux 的程序）→ unknown-elf 是裸机工具链，本课不用

安装 RuyiSDK

RuyiSDK 是面向 RISC-V 开发的工具包集合；命令入口叫 ruyi，用来查包、装工具链、查设备支持。真正把 C 代码编成机器码的是里面的 GCC（GNU 编译器）。关系可以记成：ruyi 管「装什么」，GCC 管「怎么编」。

三元组（target triple）是编译器目标平台的简写，格式大致是「CPU-厂商-系统」。本课统一用 riscv64-unknown-linux-gnu：意思是「为 64 位 RISC-V CPU、跑 Linux、用 GNU 用户态约定」生成程序。

  1. 1
    记录电脑环境
    uname -a / uname -m — 确认是 x86_64，写入个人环境清单。
  2. 2
    安装 ruyi
    按课程安装说明装好后：command -v ruyi 应给出路径；ruyi --version 记录版本号。
  3. 3
    更新软件源并查工具链
    ruyi update，再 ruyi list | grep toolchain，按说明安装 RISC-V GNU Toolchain。
  4. 4
    确认目标三元组
    riscv64-unknown-linux-gnu-gcc -dumpmachine 应打印 riscv64-unknown-linux-gnu。
# 电脑架构
$ uname -a

# ruyi 位置与版本
$ command -v ruyi
$ ruyi --version

# 更新源并查询工具链
$ ruyi update
$ ruyi list | grep toolchain

# 工具链三元组
$ riscv64-unknown-linux-gnu-gcc -dumpmachine
riscv64-unknown-linux-gnu

最小编译验证

写一个空程序，用交叉编译器编一下，再用 file 看产物是不是 RISC-V。ELF 是 Linux 上可执行文件与目标文件的常见容器格式；file 会读出里面的架构字段。

$ printf 'int main(void) { return 0; }\n' > mini.c
$ riscv64-unknown-linux-gnu-gcc mini.c -o mini-riscv
$ file mini-riscv
mini-riscv: ELF 64-bit LSB executable, UCB RISC-V, ...

在电脑上直接 ./mini-riscv 会失败——指令集不同。验收看 file 输出里有 RISC-V 字样即可。

本节成果：荔枝派 4A 接口标注 · 个人环境清单 · ruyi 路径与版本 · 三元组 riscv64-unknown-linux-gnu · 最小 RISC-V 可执行文件与 file 日志。
1.2

烧录、开发通道与一键准备/一键编译

烧录系统

烧录是把系统镜像按分区写入板子存储介质（eMMC / SD）。镜像里通常有：引导程序（上电后最先跑）、Linux 内核与设备树（描述硬件）、根文件系统（用户空间程序与配置）。

镜像发布包 ├─ 引导程序 ├─ Linux 内核与设备树 └─ RevyOS 根文件系统 ↓ 烧录（按分区写入） eMMC / SD ↓ 上电 串口出现启动日志 → 登录界面

固定流程：校验镜像哈希 → 认准目标设备号 → 严格按发布说明写入 → 工具报告成功后再上电。

三条开发通道

串口（UART）

本地调试线

UART 是按位串行收发的接口。接线：GND-GND、TX→RX、RX→TX，3.3V。不依赖网络，直接看启动日志。

SSH

远程主通道

SSH 是加密的远程终端。日常开发用它登录板子。本课用 Ed25519 密钥登录；首次连接须核对主机指纹。

SCP

文件传输

SCP 复用 SSH 连接拷文件。上传后在板端跑 sha256sum，与电脑端哈希一致即传输完整。

# 板端启用 SSH
$ sudo systemctl enable --now ssh

# 电脑生成密钥
$ ssh-keygen -t ed25519 -C "riscv-course"

# 首次连接——核对指纹后再输入 yes
$ ssh user@board-ip

# 部署公钥并免密验证
$ ssh-copy-id user@board-ip
$ ssh user@board-ip 'hostname; uname -m'
# SCP 上传并校验
$ printf 'transfer test\n' > test.txt
$ sha256sum test.txt
$ scp test.txt user@board-ip:/tmp/
$ ssh user@board-ip 'sha256sum /tmp/test.txt'
指纹变更：可能是板子重装系统，也可能连错了设备。先核对 IP 与串口里的主机指纹，再清理 known_hosts 旧记录。

1.3

交叉编译、Makefile 与 CoreMark

交叉编译与 Makefile

交叉编译：在电脑（x86_64）上编译，生成能在荔枝派 4A（riscv64）上运行的程序。编译器、汇编器、链接器都必须用面向 RISC-V 的版本——即 1.1 装好的 riscv64-unknown-linux-gnu-gcc 系列。

源代码（.c） ↓ riscv64-unknown-linux-gnu-gcc（在电脑上跑） RISC-V 可执行文件（ELF） ↓ scp 荔枝派 4A（riscv64，实际运行）

Makefile 是描述「怎么编这个工程」的脚本；敲 make 就会按规则调用编译器。把交叉前缀写进 Makefile，避免每次手敲冗长命令：

# Makefile — 交叉编译模板
CROSS_COMPILE := riscv64-unknown-linux-gnu-
CC            := $(CROSS_COMPILE)gcc
CFLAGS        := -O2 -Wall
TARGET        := app

$(TARGET): main.c
	$(CC) $(CFLAGS) main.c -o $(TARGET)

clean:
	rm -f $(TARGET)

.PHONY: clean
$ make
$ file app
app: ELF 64-bit LSB executable, UCB RISC-V, ...
常见陷阱：忘记设 CC 会默默用电脑自带的 GCC，产物在板端报 Exec format error。先用 file 查架构，再查运行问题。Makefile 命令行必须用 Tab 缩进。

构建 CoreMark

CoreMark 是测处理器算力的小基准程序，输出可比较的分数（常见为 iterations/sec）。本课用它走通「编译 → 传输 → 运行 → 读数」；分数高低不是重点，链路走通才是。

# 获取源码（地址以课程指引为准）
$ git clone https://github.com/eembc/coremark.git
$ cd coremark

# 交叉编译
$ make PORT_DIR=linux64 \
       CC=riscv64-unknown-linux-gnu-gcc \
       LD=riscv64-unknown-linux-gnu-gcc \
       CFLAGS="-O2 -DPERFORMANCE_RUN=1" \
       XCFLAGS="-DFLAGS_STR=\"cross-riscv64\""

$ file coremark.exe
coremark.exe: ELF 64-bit LSB executable, UCB RISC-V, ...

产物文件名因版本而异（常见 coremark.exe）。编好后用 SCP 传到板端、SSH 运行并记录分数——完整验收见 lab.html。

本节成果：交叉编译 Makefile 模板 · 一次可运行的 app · CoreMark 的 RISC-V 可执行文件。上板跑分见本章实验；「为什么必须交叉编译」见下一节深入理解。
深入理解

电脑编、板子跑：同一份源码为何不能直接执行

原因：电脑与荔枝派 4A 的指令集不同

你的开发电脑是一台普通 PC：CPU 按 x86_64（或少数机器上的 ARM）指令集执行机器码。荔枝派 4A 的 CPU 按 riscv64 指令集执行机器码。两边「加一条、跳一下」的二进制编码不一样——就像两种不同的机器语言。

因此：在电脑上敲普通的 gcc main.c -o app，得到的是本机架构的 ELF。这个文件只能在本机跑；拷到荔枝派 4A 上，内核会拒绝加载（常见报错是 Exec format error）。反过来，板子上编好的 RISC-V 程序拷回电脑，也一样跑不了。

要让程序在板子上跑，必须在电脑上使用交叉工具链：一套仍然跑在电脑上的编译器全家桶，但它专门生成 RISC-V 机器码，并链接面向 RISC-V Linux 的 C 库。本课通过 RuyiSDK 安装，前缀是 riscv64-unknown-linux-gnu-。

机制：同一份 main.c，两条编译路径

下面这张图把「同一份源码、两种产物」画清楚。左边是本机原生编译；右边是交叉编译。请顺着箭头读：关键在哪里分叉、产物各自能去哪里。

同一份 main.c │ ├─────────────────────┬───────────────────── │ │ ▼ ▼ 电脑自带 gcc riscv64-unknown-linux-gnu-gcc （本机工具链） （交叉工具链，经 RuyiSDK 安装） │ │ │ 生成 x86_64 │ 生成 RISC-V │ 机器指令 │ 机器指令 │ │ ▼ ▼ app-host（ELF） app-riscv（ELF） 架构：x86_64 架构：UCB RISC-V │ │ ▼ ▼ 只能在电脑 ./app-host 用 SCP 拷到荔枝派 4A 在板子上会 Exec format 再 ssh 执行 ./app-riscv error → 板子 Linux 加载并运行

读图时抓住三步。第一，编译器选错，架构就错：左边 gcc 把 C 翻成电脑 CPU 认得的指令；右边交叉 gcc 把同一份 C 翻成板子 CPU 认得的指令。第二，产物用 file 一眼可辨：输出里写 x86-64 还是 UCB RISC-V，就是验收标准。第三，传输与运行是两件事：交叉编译只解决「编对架构」；还要用 SCP 把文件送到板子，再在板端执行——板端的 Linux 负责加载这个 RISC-V ELF。

交叉工具链内部仍然是「编译 → 汇编 → 链接」，只是每一步的目标都是 RISC-V：编译器产出 RISC-V 汇编，汇编器变成 .o，链接器再链上 RISC-V 版的 C 运行库（sysroot）。调用约定、系统调用号都按板子来，这样程序在板端跑起来才像「正常 Linux 程序」。

交叉编译内部（都在电脑上执行） main.c ↓ 编译器：C → RISC-V 汇编 main.s ↓ 汇编器：汇编 → 目标文件 main.o ↓ 链接器：.o + RISC-V C 库 → 可执行文件 app-riscv（ELF，riscv64） ↓ scp 荔枝派 4A：./app-riscv

三元组 riscv64-unknown-linux-gnu 里每一段都在回答「编给谁」：CPU 是 riscv64、系统是 Linux、用户态约定是 GNU。选错成 unknown-elf 会得到裸机工具链，不适合本课跑 Linux 用户程序的场景。

学生如何亲自验证这条链

不要只背结论——用同一份 main.c 走两遍，对照 file 输出，再决定能不能上板。

  1. 1
    写同一份源码
    printf 'int main(void) { return 0; }\n' > main.c
  2. 2
    本机路径
    gcc main.c -o app-host，再 file app-host。应看到 x86-64（或你电脑对应的本机架构）。在电脑上 ./app-host 能跑。
  3. 3
    交叉路径
    riscv64-unknown-linux-gnu-gcc main.c -o app-riscv，再 file app-riscv。应看到 UCB RISC-V。在电脑上 ./app-riscv 应失败。
  4. 4
    上板
    scp app-riscv user@board-ip:~/，再 ssh user@board-ip './app-riscv'。板端应能执行；若报 Exec format error，先核对 file 是否真是 RISC-V。
$ gcc main.c -o app-host
$ file app-host
# … ELF … x86-64 …

$ riscv64-unknown-linux-gnu-gcc main.c -o app-riscv
$ file app-riscv
# … ELF … UCB RISC-V …

$ scp app-riscv user@board-ip:~/
$ ssh user@board-ip './app-riscv; echo exit:$?'

整条故事可以收成一句话：指令集不同 → 本机 gcc 编不出板子能跑的二进制 → 经 RuyiSDK 安装的交叉工具链专门生成 RISC-V Linux ELF → file 确认架构 → SCP 上板 → 板端执行。 CoreMark 只是把这条链换成更大一点的工程；验收逻辑不变。

本节成果：能口述指令集 / 交叉编译 / 三元组；能画「同一 main.c 两条路」；能用 file + SCP 亲自验证。带着这条链进入本章实验跑 CoreMark。
