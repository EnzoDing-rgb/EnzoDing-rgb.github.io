对应讲义 2.1–2.3 · 脚手架 chapters/ch02/code/fake-thermostat/

说明

讲义 2.1–2.3 已覆盖类型、函数、指针与结构体。本实验给你一份故意埋了 bug 的假温控程序：没有真实 DHT 传感器，温度来自内置数组或标准输入。你的任务是按关卡表把逻辑改对，直到输出与期望序列完全一致。

  • 脚手架目录：chapters/ch02/code/fake-thermostat/（半成品主要在 src/thermo.c，接口声明在 include/thermo.h，驱动入口在 src/main.c）
  • 输入：默认跑内置温度序列；也可用 --stdin 从标准输入读浮点数（负数表示本轮「读失败」）
  • 输出：每一步打印温度与风扇 ON/OFF 决策；读失败时应打印 READ_FAIL 并保持上一风扇状态
前置：已读完讲义 2.1–2.3；本机装有 gcc 与 make。本章用本机编译，专注改 C 逻辑，不必上板。
不要一上来大段重写。先跑通半成品，对照关卡表，一次只改一类 bug。脚手架内 README.md 与源码注释里的 L1–L4 标记可对照使用。

步骤

1. 构建与首次运行（先看坏现象）

进入脚手架目录，编译并运行。半成品会编译通过，但决策是错的——这是故意的，用来对照关卡表找坑。

$ cd chapters/ch02/code/fake-thermostat
$ make
$ ./build/fake-thermo
# 观察内置序列下的温度与风扇输出，记下「哪里不对」

# 也可用 stdin 喂温度（一行一个浮点数；负数表示读失败）
$ printf '25\n27\n29\n27\n24\n-1\n' | ./build/fake-thermo --stdin

本步成功判据：能编译出 ./build/fake-thermo 并看到每步输出；你已确认当前行为与下方「修好后期望表」不一致（说明坑还在）。

2. 按关卡改代码（一次一类）

滞回控温的含义：温度高于上限才开风扇，低于下限才关风扇；夹在中间时保持上一状态，避免在阈值附近频繁开关。本实验默认配置：t_high = 28.0，t_low = 26.0，初始风扇为 OFF。

主要改 src/thermo.c 里的 thermo_decide 与 thermo_read。每修完一关，重新 make 再跑，对照现象是否消失。

关卡修好前你会看到什么你要改成什么（成功判据）主要坑类型
L1 温度升高时风扇反而关，或该开不开、该关不关 比较方向正确：temp > t_high → 风扇 ON；temp < t_low → 风扇 OFF；中间保持 错误比较
L2 阈值「怎么改都不对劲」，行为像拿错了结构体里的另一个字段 决策只使用 cfg->t_high 与 cfg->t_low，不要误用 period_ms（采样周期毫秒数，不是温度阈值） 结构体字段用错
L3 传入空配置指针或空风扇状态指针时崩溃，或出现未定义行为 thermo_decide / thermo_read 在指针为 NULL 时安全返回，不解引用空指针 指针使用
L4 传感器读失败时仍用垃圾温度做决策，或失败路径却返回「成功」 thermo_read 失败时返回非 0 且不写输出；主循环侧打印错误并跳过本轮决策，保持上一风扇状态 缺少错误分支

修 L3 时，可用讲义练习或在测试代码里对 thermo_decide(NULL, …) / thermo_read(NULL) 调用一次，确认进程不崩溃。修 L4 时，内置序列最后一步会注入「读失败」，输出应出现 READ_FAIL。

3. 对照内置序列做最终验收

四关都修好后，直接运行 ./build/fake-thermo（默认内置序列）。期望决策如下（初始风扇 OFF）：

步温度 °C期望风扇说明
125.0OFF低于下限，保持关
227.0OFF落在滞回带中间，保持上一状态
329.0ON高于上限，打开
427.0ON仍在中间带，保持开
524.0OFF低于下限，关闭
6读失败（哨兵）保持 OFF打印 READ_FAIL（或等价错误日志），不根据垃圾温度改风扇

本步成功判据：六步输出与上表一致；你能口头指出半成品里至少三类坑（比较、字段、指针、错误分支中任选）。
