2.1

类型、变量、控制流

类型与变量

程序处理的是有类型的数据。类型规定一块内存按什么规则解释（整数、小数、文字等）。本课最常见的几种：

类型典型用途例子
int计数、状态、返回码风扇开关 0/1、循环下标
float / double温度、小数28.5 °C
char / 字符串日志文字"FAN ON"
int 返回值成功 / 失败0 成功，非 0 失败
int fan_on = 0;          /* 0 = 关，1 = 开 */
float temp_c = 27.5f;    /* 当前温度 */
const float T_HIGH = 28.0f;
const float T_LOW  = 26.0f;
习惯：配置常量用 const 写清含义；开关语义用 0/1，并在注释里标明「开/关」。

控制流：判断与循环

控制流决定程序「先做什么、再做什么」——分支、循环都算。

温控里会用到滞回：开和关用两个不同的阈值，中间一段「保持上一状态」，避免温度在临界附近来回抖动时风扇狂开狂关。例如：温度严格大于 28 °C 才开风扇，严格小于 26 °C 才关；落在 26–28 之间则不改当前开关状态。

if (temp_c > T_HIGH) {
    fan_on = 1;
} else if (temp_c < T_LOW) {
    fan_on = 0;
}
/* 落在 [T_LOW, T_HIGH] 之间：保持现状（滞回） */

周期性采样用循环：

for (int i = 0; i < n; i++) {
    /* 读一次温度，做一次决策，打印一行 */
}

while (running) {
    /* 主循环：采样 → 决策 → 延时 */
}
温度序列示例（滞回） T_HIGH = 28 T_LOW = 26 25 → 关 27 → 保持 29 → 开 27 → 保持 25 → 关 陷阱：比较方向写反、用错阈值字段，都会让「该开不开 / 该关不关」。

读图：从 25 升到 27 仍关着（还没越过 28）；到 29 才开；再降到 27 仍保持开（还没低于 26）；到 25 才关。中间「保持」就是滞回在起作用。

跟做与练习

  1. 写出：温度严格大于 28 开风扇，严格小于 26 关风扇，中间保持原状态。
  2. temp >= T_HIGH 与 temp > T_HIGH 在边界 28.0 上行为差在哪里？
  3. 若把 T_HIGH 和 T_LOW 写反，会出现什么现象？

2.2

函数、头文件与多文件工程

函数：把「一件事」包起来

函数是一段有名字的代码块，有参数、返回值和函数体。本课常见写法：

/* 返回 0 成功，非 0 失败 */
int read_temp(float *out_c);

void apply_fan_policy(float temp_c, int *fan_on);

调用方根据返回值决定是否继续用读到的温度：

float t;
if (read_temp(&t) != 0) {
    /* 读失败：打日志，不要用未初始化的 t */
    return;
}
apply_fan_policy(t, &fan_on);

头文件：声明接口

头文件（.h）告诉其他 .c：「这些函数/类型存在」。实现放在对应的 .c 里。用 include guard 防止重复包含：

/* greet.h */
#ifndef GREET_H
#define GREET_H

void greet(const char *name);

#endif

声明在 .h

函数原型、结构体类型、对外宏——给调用方看。

定义在 .c

函数体、静态内部变量——实现细节。

多文件工程：跟做模板

  • chapters/ch02/code/hello/ — 单文件入门
  • chapters/ch02/code/project-template/ — src/ + include/ + build/
project-template/ ├── Makefile ├── include/greet.h ← 接口声明 ├── src/main.c ← 调用 greet() ├── src/greet.c ← 实现 greet() └── build/ ← make 生成，可删
$ cd chapters/ch02/code/project-template

  2. 为什么调用方 #include "greet.h"，而不是把函数体复制进 main.c？

2.3

指针、数组、结构体

指针与数组

指针是「存着某个对象地址」的变量。函数要写回调用方的变量时，就传指针：

void set_fan(int *fan_on, int value)
{
    if (fan_on == NULL)
        return;
    *fan_on = value;   /* 改的是调用方那份变量 */
}

数组是连续同类型元素。假温度序列常用数组驱动：

static const float seq[] = { 25.0f, 27.0f, 29.0f, 27.0f, 24.0f };
for (size_t i = 0; i < sizeof(seq)/sizeof(seq[0]); i++) {
    /* 用 seq[i] 当本轮温度 */
}
解引用前先确认指针非 NULL；数组下标不要越界。闯关脚手架里故意埋了相关坑。

结构体：把相关字段捆在一起

结构体把一组相关字段捆成一种类型，配置不必散落成一堆全局变量：

typedef struct {
    float t_high;      /* 高于此开风扇 */
    float t_low;       /* 低于此关风扇 */
    int   period_ms;   /* 采样周期（毫秒） */
} ThermoConfig;

void decide(const ThermoConfig *cfg, float temp, int *fan_on)
{
    if (temp > cfg->t_high)
        *fan_on = 1;
    else if (temp < cfg->t_low)
        *fan_on = 0;
}

读结构体字段用 cfg->t_high（指针）或 cfg.t_high（变量本身）。用错字段名是真实 bug 的常见来源。

  1. 1
    看头文件 — 有哪些结构体、API、返回值约定。
  2. 2
    看 main — 输入从哪来、如何循环、打印什么。
  3. 3
    看实现 — 比较方向、字段名、空指针、错误分支是否完整。
  4. 4
    对照行为表 — 改一处、跑一次，输出是否逐步贴近验收表。
本节成果：能读指针写回、数组序列与配置结构体。下一节深入理解讲「多个 .c 如何变成一个可执行文件」。
深入理解

从 .c 到可执行文件：编译、链接，以及 main 之前

原因与机制：先编译成 .o，再链接成可执行文件

你在工程里写了多个 .c 文件，但最终只得到一个可执行文件。这不是「把几个源文件粘在一起」，而是分两步：编译和链接。

编译：把每个 .c 翻译成对应的目标文件 .o。目标文件里已经有该文件函数的机器码，以及一张「符号表」——记录「我提供了哪些函数名 / 我还引用了哪些别处定义的名字」。

链接：把多个 .o 以及 C 运行库拼成一个 ELF。链接器负责解析未定义符号：例如 main.o 调用了 greet，就要在 greet.o 里找到定义；找不到就报 undefined reference。

编译 + 链接 main.c ──编译──► main.o （机器码 + 符号：提供 main，需要 greet） greet.c ──编译──► greet.o （机器码 + 符号：提供 greet） │ ▼ 链接（ld，或由 gcc 调用） 解析未定义符号：main 需要的 greet ← 来自 greet.o 再链上 C 运行库（printf、启动代码等） │ ▼ build/app（可执行 ELF）

读图：左边两行是「各自独立编译」——改 greet.c 通常只需重编 greet.o，不必重编 main.c。中间箭头是链接：把分散的机器码和符号对上号。右边是最终产物。头文件 .h 不参加链接；它只在编译阶段把声明抄进需要它的 .c，让编译器知道「greet 长什么样」。

继续：你写的 main 并不是进程真正的第一步

对你写的业务代码来说，入口是 main。对整个进程来说，操作系统加载 ELF 后，先跑的是 C 运行库提供的启动代码，常记作 _start。

进程启动（简化） 操作系统加载 ELF │ ▼ _start（运行库提供，不是你写的） ├─ 准备栈（stack）：函数调用时放局部变量与返回地址 ├─ 准备参数：argc、argv（命令行参数个数与字符串数组） ├─ 准备环境变量指针 ├─ 做全局 / 静态变量的初始化（含构造相关工作） │ ▼ 调用 main(argc, argv) │ ▼ main 返回后，运行库做收尾（如冲刷缓冲）并退出进程

读图：栈让函数调用有地方放临时数据；argc / argv 让 main 能读到你在命令行敲的参数；全局初始化保证你在 main 里读到的全局变量已经是源码里写好的初值。然后才进入你熟悉的 main。所以「程序从 main 开始」是应用层说法；完整链条是 加载 → _start → 栈与参数与全局初始化 → main。

学生如何亲自验证

用 project-template 走一遍，把抽象步骤变成终端证据：

  1. 1
    看中间产物
    在 project-template 里 make CROSS_COMPILE=，然后列出 build/：应能看到 .o 与最终 app。用 file build/*.o build/app 确认目标文件与可执行文件的角色不同。
  2. 2
    制造链接错误
    临时把 greet.c 里的函数名改错再 make，应看到 undefined reference to greet——这就是「编译过了、链接对不上」。
  3. 3
    确认入口不是只有 main
    对可执行文件跑 nm build/app | grep -E '_start|main'（若本机有 nm）：通常能同时看到 main 与启动相关符号。不必背所有符号名，目的是意识到 ELF 里不止你写的函数。
  4. 4
    对照实验工程
    闯关工程 fake-thermostat/ 同样是多 .c → 多 .o → 一个可执行文件。改 thermo.c 再 make，链接器会把新的 thermo.o 和 main.o 重新拼在一起。
$ cd chapters/ch02/code/project-template
$ make CROSS_COMPILE=
$ ls build/
$ file build/*.o build/app
$ nm build/app | grep -E '_start|main' || true
$ ./build/app

整条故事：.c → 编译 → .o → 链接 → 可执行文件；运行时先 _start（栈、argv、全局初始化），再进 main。 读懂这条链，后面改温控逻辑时就知道「改哪个文件、为什么重新 make 就够了」。
