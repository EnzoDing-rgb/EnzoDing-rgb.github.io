说明

讲义 6.1–6.3 已说明：单线程里采样、控风扇、跑 MQTT 会互相堵住；拆成三条线程后，共享的温度/风扇状态若不加锁会出现竞态（多线程同时读写同一变量，结果不可预期）。本实验交付两份工程，顺序固定：先在电脑上看清「丢更新」与 ThreadSanitizer 报告，再在荔枝派 4A 上跑产品骨架并完成无锁→加锁对照。

  • chapters/ch06/code/race-demo/ — 在电脑上跑的强制竞态小程序；配合主机 gcc -fsanitize=thread（ThreadSanitizer，简称 TSan）指出谁在抢同一变量
  • chapters/ch06/code/tri-thread/ — 板上产品骨架：采集线程写温度、控制线程按阈值开/关风扇、通信线程跑 MQTT
前置：已读讲义 6.1–6.3；第五章控灯与 Broker 通路可用；温控相关接线可用。工具链分工：板上程序用 RuyiSDK 交叉编译；TSan 只用主机本机 gcc（交叉工具链通常不开 TSan）。
race-demo 故意无锁，用来制造竞态；不要把无锁写法抄进 tri-thread 的产品路径。板上验收以加锁版本（USE_LOCK 1）为准。

步骤

A. 电脑：race-demo + TSan（先做，固定顺序）

这部分不需要板子。目的：用一个「很多线程同时给同一计数器加一」的小程序，亲眼看到最终计数偏小（丢更新），再用 TSan 指出具体是哪两行源码在抢。

  1. 无锁跑一遍，记录最终计数（应明显小于「线程数 × 每线程循环次数」的期望值）： 
    $ cd chapters/ch06/code/race-demo
$ make
$ ./race-demo
# 把打印的最终计数与期望值抄下来，证明丢更新
  2. 固定必做：在跑带 TSan 的版本之前，先执行： 
    $ sudo sysctl vm.mmap_rnd_bits=28
    为什么要做这一步：把内核的用户态地址空间随机化位数调到 ThreadSanitizer 当前工具链能稳定配合的范围；否则本机上 TSan 可能直接启动失败或行为异常，导致你看不到数据竞争报告。这是本实验的固定步骤，不是可跳过项。
  3. 打开 TSan 再跑，保存完整报告： 
    $ make tsan
$ ./race-demo
# 终端会出现 ThreadSanitizer 的 data race 报告；全文保存
    在报告里标出它指出的两处源码访问行（通常是两个线程对同一全局计数器的读写），对照源码确认。

本段成功判据：有「计数偏小」的记录；有完整 TSan 报告；能指出抢变量的两行。

B. 板上：tri-thread（无锁必现 → 加锁修好 → 干净停）

  1. 配置与补全三线程
    打开 chapters/ch06/code/tri-thread/，按接线表与 Broker 地址改 GPIO / Broker 相关宏。补全三条线程的循环职责：
    • 采集线程：周期读温度，写入共享状态；
    • 控制线程：读共享温度，按阈值开/关风扇；
    • 通信线程：跑 MQTT 循环（下行命令 / 上行状态，主题约定与第五章一致）。
  2. 先不加锁（脚手架提供 USE_LOCK 0）
    编译部署后跑一段时间（建议至少 1–2 分钟，并在此期间用 MQTT 发几次命令或观察多轮采样）。必须记录错乱现象——下面是具体、可写入实验报告的例子（出现一类即可，多记更好）：
    • 日志里温度在相邻两行之间无合理解释地大幅跳变（例如刚打印 27.1，下一行突然变成 0.0 或跳到 45，随后又跳回），而传感器并未被那样折腾；
    • 风扇状态与温度矛盾：温度已明显高于 T_HIGH，日志却长时间显示 fan=OFF；或温度已低于 T_LOW，风扇却仍显示 ON；
    • 同一时刻附近的日志互相打架：采集线程打印的温度与控制线程依据的温度明显不是同一份「当前值」；
    • MQTT 上报的状态与板端本地日志不一致（例如上行说灯/风扇 ON，本地日志却是 OFF）。
    把原始日志片段粘贴进记录，不要只写「有点乱」三个字。
  3. 改为 USE_LOCK 1
    所有对共享温度、风扇状态等字段的读写都放进同一把锁的加锁/解锁区间。重新编译部署，确认：阈值行为恢复正常；先前记录的那类错乱不再复现。
  4. 交叉编译部署命令 
    $ cd chapters/ch06/code/tri-thread
$ make clean && make
$ file tri-thread
# 应为 RISC-V ELF
$ scp tri-thread user@board-ip:~/
$ ssh user@board-ip './tri-thread'
  5. 功能验证
    风扇按阈值动作（超温开 / 降温关）；MQTT 下行或上行至少一项可演示（与第五章主题约定一致）。采集、控制、通信三条线的日志或行为均可观察。
  6. 干净停
    在板端按 Ctrl+C：期望风扇关断（安全落到 OFF），然后进程退出。不要留下风扇空转、进程僵死。
