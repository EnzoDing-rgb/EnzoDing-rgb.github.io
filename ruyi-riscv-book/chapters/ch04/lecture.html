4.1

串口是聊天线：字节怎么进出

学习目标

串口直觉

把串口当成「打字进、回显出」的双向管道

字节流

一行命令就是一串字符，以换行结束

读写 API

用 read / write（或 stdin/stdout）收发

与控制台关系

SSH 终端也是同一套「读一行、写一行」逻辑

字节怎么进出

串口（UART 接到电脑后，常表现为 USB 转串口工具里的黑窗口）按约定波特率逐字节收发。对本课程序来说：你敲的字变成字节流进入进程；程序 write 回去的字出现在同一窗口。

你（键盘） │ 敲 "status\n" ▼ 终端 / 串口工具 ──► stdin 或 /dev/tty* │ read() ▼ 你的 C 程序（解析命令、做事） │ write() / printf ▼ 同一窗口显示回复

第一章用串口登录系统；本章用同一通道做应用层对话：程序读你的命令行，再打印状态。日常验收也可用 SSH 终端的 stdin/stdout——读写逻辑相同，脚手架默认 USE_STDIO=1 即走这条路。

说法含义本课怎么用
字节流没有「消息边界」；你自己约定「一行一条命令」读到 \n 再解析
阻塞读read 一直等到有数据才返回4.1 可接受；4.3 要用 select 同时盯多路
回显把收到的字再打回去，确认通路第三章 uart-echo 已练过

用终端练读写

先不接第二路硬件串口：在 SSH 里跑程序，从 stdin 读一行、往 stdout 写回复。确认「打字 → 程序 → 回复」闭环后，再把同一套解析接到真实 /dev/tty*（宏 USE_STDIO=0）。

/* 读一行（简化示意） */
char line[128];
if (fgets(line, sizeof(line), stdin)) {
    /* 去掉末尾 \n，再查命令表 */
}
注意：真串口时 TX↔RX 交叉、GND 共地、波特率一致、电平 3.3 V。不要占用正在当系统控制台的那一路，除非课程接线表明确指定。

课堂练习

  1. 用一句话说明：串口窗口里看到的文字，和程序里的 char 数组是什么关系？
  2. 若只 read 一个字节就解析命令，会出现什么问题？
本节成果：能口述字节进出路径 · 能用 stdin 读一行命令 · 知道命令以换行分界。

读 DHT22 + 串口命令：status / set

学习目标

复用传感执行

命令表

字符串 → 处理函数，便于扩展

status

打印温度、湿度、风扇、阈值

set

现场改 T_HIGH / T_LOW

命令表：字符串对应函数

命令表是一张「名字 → 处理函数」的表。收到一行字后，先拆出命令名，再查表调用。比一长串 if/else 更清晰，后面加 help 也只多一行。

输入行："set high 30\n" │ ▼ 拆词：cmd="set", arg1="high", arg2="30" │ ▼ 查命令表 → 调用 cmd_set(...) │ ▼ 改 t_high = 30.0，打印确认
struct cmd_entry {
    const char *name;
    int (*handler)(char *args);  /* 返回 0 成功 */
};

status 与 set

命令例子行为
statusstatus打印最近温度、湿度、风扇 ON/OFF、t_high/t_low
set high <数>set high 30改上限阈值并确认打印
set low <数>set low 25改下限阈值；须保持 low < high

DHT22 仍按第三章方式周期采样；读失败打 [ERR]，程序继续跑。串口命令只改内存里的阈值变量，立刻影响下一轮决策，不必重启进程。

4.3

两边同时工作 · 深入理解：select 同时等采样与串口输入

学习目标

问题

阻塞读串口时，采样会卡住；死循环 sleep 时，打字要等

select

一次等待「多个文件描述符 / 超时」

双源主循环

超时到 → 采样控温；stdin 可读 → 解析命令

深入理解

亲手看见两边都能及时响应

为何不能「先读串口再采样」串成一条线

若主循环写成「fgets 等你打字 → 再读 DHT → 再 sleep」：不打字时风扇永远不更新。若写成「采样 → sleep 2 秒 → 再看一眼串口」：你敲完字最多等 2 秒才有回复，体感像卡死。两边都要及时，就需要同时等。

错误 1：阻塞读命令 等打字 ……（风扇停更） 错误 2：固定 sleep 再轮询 采样 → sleep 2s → 才读串口（打字延迟大） 正确：select 同时等：stdin 可读 OR 采样超时到 │ │ ▼ ▼ 解析命令 读 DHT + 阈值控风扇

深入理解：select 在等什么

select 是系统调用：告诉内核「这些文件描述符里，有谁可读/可写了，或者超时了，再叫醒我」。本课用法：

  1. 把串口（或 stdin）放进读集合；
  2. 超时设成「距下次采样还剩多少毫秒」；
  3. 返回后：若描述符可读 → 读一行、查命令表；若因超时返回 → 执行一次采样与控温。
/* 伪代码：主循环骨架（实验里补全） */
for (;;) {
    fd_set rfds;
    FD_ZERO(&rfds);
    FD_SET(cmd_fd, &rfds);
    struct timeval tv = { .tv_sec = 0, .tv_usec = remain_us };

    int n = select(cmd_fd + 1, &rfds, NULL, NULL, &tv);
    if (n < 0) { perror("select"); break; }

    if (n == 0) {
        /* 超时：该采样了 */
        sample_and_control();
        remain_us = SAMPLE_MS * 1000;
    } else if (FD_ISSET(cmd_fd, &rfds)) {
        /* 有输入：读一行并处理命令 */
        handle_command_line();
        /* 注意：处理命令后仍要扣减剩余等待时间 */
    }
}
深入理解要看到的现象：程序在跑自动控温的同时，你敲 status 立刻有回复；敲 set high 30 后下一轮采样按新阈值动作。这证明两路事件都被 select 叫醒，而不是互相堵死。
