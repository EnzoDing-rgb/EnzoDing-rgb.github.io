4.1

串口是聊天线：字节怎么进出

字节怎么进出

串口是一种按约定波特率、逐字节收发的双向通信通道。硬件上常叫 UART；接到电脑后，往往表现为 USB 转串口工具里的黑窗口，或系统里的设备节点 /dev/tty*。对本课程序来说：你敲的字变成字节流进入进程；程序 write / printf 回去的字出现在同一窗口。

你（键盘） │ 敲 "status\n" ▼ 终端 / 串口工具 ──► stdin 或 /dev/tty* │ read() / fgets() ▼ 你的 C 程序（解析命令、做事） │ write() / printf ▼ 同一窗口显示回复

第一章用串口登录系统；本章用同一类通道做应用层对话：程序读你的命令行，再打印状态。日常验收也可用 SSH 终端的 stdin/stdout——读写逻辑相同，脚手架默认 USE_STDIO=1 即走这条路。

说法含义本课怎么用
字节流没有天然的「消息边界」；你自己约定「一行一条命令」读到 \n 再解析
阻塞读read / fgets 一直等到有数据才返回4.1 练通路时可用；两边同时工作时要用 select
回显把收到的字再打回去，确认通路通畅先确认「打字进、回复出」，再接命令表

用终端练读写

先不接第二路硬件串口：在 SSH 里跑程序，从 stdin 读一行、往 stdout 写回复。确认「打字 → 程序 → 回复」闭环后，再把同一套解析接到真实 /dev/tty*（宏 USE_STDIO=0）。

/* 读一行（简化示意） */
char line[128];
if (fgets(line, sizeof(line), stdin)) {
    /* 去掉末尾 \n，再查命令表 */
}
注意：真串口时 TX↔RX 交叉、GND 共地、波特率一致、电平 3.3 V。不要占用正在当系统控制台的那一路，除非课程接线表明确指定。
本节成果：能口述字节进出路径 · 能用 stdin 读一行命令 · 知道命令以换行分界。
4.2

读 DHT22 与串口命令：status / set

DHT22 是温湿度探头：一根数据线接 GPIO，按协议周期上报温度和湿度。第三章已用它做温控闭环；本章继续周期采样，并在串口侧增加查状态、改阈值的命令。

命令表：字符串对应函数

命令表是一张「名字 → 处理函数」的表。收到一行字后，先拆出命令名，再查表调用。比一长串 if/else 更清晰，后面加 help 也只多一行。

输入行："set high 30\n" │ ▼ 拆词：cmd="set", arg1="high", arg2="30" │ ▼ 查命令表 → 调用 cmd_set(...) │ ▼ 改 t_high = 30.0，打印确认
struct cmd_entry {
    const char *name;
    int (*handler)(char *args);  /* 返回 0 成功 */
};

status 与 set

命令例子行为
statusstatus打印最近温度、湿度、风扇 ON/OFF、t_high/t_low
set high <数>set high 30改上限阈值并确认打印

DHT22 仍按第三章方式周期采样；读失败打 [ERR]，程序继续跑。串口命令只改内存里的阈值变量，立刻影响下一轮决策，不必重启进程。

两边同时工作

为何「先读串口再采样」串成一条线不行

本章程序要同时完成两件事：约每 2 秒读一次 DHT22 并按阈值开/关风扇；随时响应你敲的 status / set。若主循环写成「等你打完字再采样」或「固定 sleep 再瞥一眼串口」，总有一边会挨饿。下一节「深入理解」把两种错误写法、正确的 select 机制，以及如何验收讲完整。

深入理解

select：同时等串口输入与采样超时

这一节按「错误写法 → 正确机制 → 返回值含义 → 剩余时间怎么扣 → 怎么验收」把链路讲完。先记住三个词：串口是字节双向通道（见 4.1）；DHT22 是周期上报温湿度的探头（见 4.2）；select 是系统调用——你告诉内核「这些文件描述符里谁就绪了，或者超时到了，再叫醒我」。

两种错误写法：为什么会堵死一边

错误一：主循环阻塞在串口读上。写成「先 fgets / read 等到一行命令，再读 DHT22，再控风扇」。没有人打字时，进程一直卡在读调用里，采样永远不会执行，风扇停在上次状态——温度已经越界，继电器也不会动。

错误二：只轮询传感器，打字要等固定 sleep。写成「采样 → sleep(2) → 才看一眼串口」。风扇节奏对了，但你敲完 status 最多要等满整个 sleep 才有回复，体感像卡死；若 sleep 期间有多行输入，还容易积压在缓冲里。

错误 1：阻塞读命令 等打字 ……（风扇停更，控温失效） 错误 2：固定 sleep 再轮询 采样 → sleep 2s → 才读串口（打字延迟大） 正确目标：同一时刻「盯」两件事 · 串口（或 stdin）是否可读 · 距下次采样是否已到时间

图说明：单线程里不能「先干完 A 再干 B」地串死；必须让内核在「有字可读」和「采样周期到了」之间择一叫醒你，两边都有机会马上处理。

正确机制：select 等串口 fd + 采样超时

做法是：把命令输入的文件描述符（stdin 或打开的 /dev/tty*）放进 select 的读集合；同时把超时设成「距下次采样还剩多少时间」。内核替你睡到这两种事件之一发生，再返回。

每次进入 select 之前： 读集合：cmd_fd（串口 / stdin） 超时 tv：remain_us（距下次采样的剩余微秒） select(...) │ ┌───────────┼───────────┐ ▼ ▼ ▼ 返回 < 0 返回 0 返回 > 0 出错 超时到 有描述符就绪 │ │ │ ▼ ▼ ▼ perror 采样+控温 若 FD_ISSET(cmd_fd) 重置周期 读一行、查命令表

图说明：一次 select 只回答「这次是因为超时醒的，还是因为串口有数据醒的」。你根据返回值分支，而不是自己猜。

select 返回值含义本课该做什么
< 0出错（常见被信号打断等）perror，按策略退出或重试
0超时：没有任何描述符就绪，但时间到了执行一次「读 DHT22 + 阈值判风扇」；把剩余时间重置为完整采样周期
> 0有就绪描述符；用 FD_ISSET 判断是谁若命令 fd 可读：读一行、查命令表、回复；然后扣减已经流逝的等待时间，再进下一轮 select
/* 伪代码：主循环骨架（实验里补全） */
for (;;) {
    fd_set rfds;
    FD_ZERO(&rfds);
    FD_SET(cmd_fd, &rfds);
    struct timeval tv = { .tv_sec = 0, .tv_usec = remain_us };

    int n = select(cmd_fd + 1, &rfds, NULL, NULL, &tv);
    if (n < 0) { perror("select"); break; }

    if (n == 0) {
        /* 超时：该采样了 */
        sample_and_control();
        remain_us = SAMPLE_MS * 1000;
    } else if (FD_ISSET(cmd_fd, &rfds)) {
        /* 有输入：读一行并处理命令 */
        handle_command_line();
        /* 处理完后要按墙上时钟扣减 remain_us，避免采样被拖慢 */
    }
}

剩余时间（remain）怎么工作

select 的超时参数在返回后可能被内核改写为「还剩多少没等到」——具体是否改写因平台而异，所以本课更稳妥的做法是：自己用时钟记下「本轮采样 deadline」，每次进 select 前算 remain = deadline - now。若中间因为处理了一条命令而返回，deadline 不变，只是下一次 remain 变短；等 remain 耗尽或返回 0，再采样并设定下一个 deadline。这样采样周期不会因为你敲了几行命令就整体往后滑。

时间轴示意（采样周期 2s）： t=0.0 采样，设 deadline=2.0 t=0.0 select(remain≈2.0s) t=0.7 你敲 status → select 返回 >0 → 立刻回复 再 select(remain≈1.3s) ← 没重置成 2s t=2.0 select 返回 0 → 采样，设 deadline=4.0 ...

图说明：处理命令不等于「采样时钟重新开始」。只有超时分支（或 remain 耗尽）才推进下一轮采样。

验证：风扇自动控，status / set 立刻响应

在荔枝派 4A 上跑补全后的程序，按下面顺序验收因果链是否闭合：

  1. 1
    看自动控温仍在
    不敲任何命令，改变探头附近温度（或临时用仿真温度），风扇仍应按阈值开/关。说明超时分支在跑，没有被串口阻塞吃掉。
  2. 2
    敲 status
    应立刻打印最近温湿度、风扇与阈值。说明读集合分支在跑，没有卡在固定 sleep 上。
  3. 3
    敲 set high 30（或课堂指定值）
    立刻看到确认；下一轮采样起，新阈值参与决策。说明命令改的是内存状态，且采样循环仍在继续。
验收话术：「我没有阻塞在串口读上，也没有用固定 sleep 饿死打字；我用 select 同时等串口可读和采样超时。返回 0 就采样控风扇，返回正数且 fd 就绪就处理命令，并用剩余时间保持采样节奏。所以风扇仍自动控，status/set 立刻有回复。」
