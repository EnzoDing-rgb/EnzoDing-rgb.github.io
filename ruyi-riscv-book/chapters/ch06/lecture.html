6.1

一条线会互相堵

阻塞从哪来（简述）

第五章的 mqtt-led 用网络循环一直处理 MQTT 事件——这很好，但同一条主线若还要每几秒读 DHT22、判阈值、开风扇，就会互相抢时间：网络处理慢了，采样推迟；采样或 GPIO 耗时长了，MQTT 心跳可能超时。某一段一堵，后面全停。

单线程硬塞在一起： while (1) { mqtt_处理一会儿(); // 可能卡住 读_DHT22(); // 也可能卡住 控风扇(); } 结果：某一段一堵，后面全停。

线程让进程里同时跑多段逻辑：采集线专心采样，控制线专心判阈值，通信线专心 MQTT。它们通过共享内存里的状态结构交换数据——下一节定义这份结构，再下一节处理「同时改会写乱」的问题。

和第四章 select 的关系：select 适合「一个线程盯几个 I/O」；当业务块变多（传感时序 + 网络库自带循环），用线程按职责切开更清晰。两种手段都要会。
本节成果：能解释单线程互相堵的现象 · 明确本章用三线程拆分的动机。
6.2

三线：采集、控制、通信

本节标题里的三线，完整说法是：采集线程、控制线程、通信线程。它们在同一进程里并行跑，通过一份共享状态交换数据。

职责拆分

┌─────────────────┐ 写 last_temp ┌─────────────────┐ │ 采集线程 │ ─────────────► │ │ │ 周期读 DHT22 │ │ 共享状态 │ └─────────────────┘ │ + 互斥锁 │ │ │ ┌─────────────────┐ 读温度/写风扇 │ │ │ 控制线程 │ ◄────────────► │ │ │ 按阈值开/关风扇 │ │ │ └─────────────────┘ │ │ │ │ ┌─────────────────┐ 改阈值/发状态 │ │ │ 通信线程 │ ◄────────────► │ │ │ 跑 MQTT 循环 │ └─────────────────┘ └─────────────────┘

图说明：采集只负责把最新温湿度写进共享状态；控制只负责读温度与阈值、驱动风扇；通信只负责 MQTT 下行改阈值/控灯与上行报状态。谁都不包办别人的活。

线程做什么不做什么
采集线程按周期读传感器，更新 last_temp/last_hum不直接开风扇、不跑网络库
控制线程读温度与阈值，调用风扇 GPIO不阻塞在 MQTT
通信线程跑 MQTT 循环；下行改阈值或灯；上行发布状态不做 DHT 位带采样

骨架一览

chapters/ch06/code/tri-thread/ 给出 pthread 创建、共享结构、三线程函数空壳与信号退出标志。实验中补全各线程循环与加锁读写。

struct shared_state {
    pthread_mutex_t lock;   /* 互斥锁：同一时刻只许一条线改关键字段 */
    float last_temp;
    float t_high, t_low;
    int fan_on;
    volatile int running;   /* Ctrl+C 置 0，各线程退出 */
};
$ cd chapters/ch06/code/tri-thread
$ make clean && make
$ file tri-thread
$ scp tri-thread user@board-ip:~/
线程函数必须是 void *(*)(void *)；创建后用 pthread_join 等其结束。风扇与 GPIO 初始化放在开线程之前，避免多线同时申请同一 line。
本节成果：能画出采集、控制、通信三线与共享状态 · 能编译部署 tri-thread 骨架 · 知道各线程边界。
6.3

共享状态与干净停

锁与干净停

采集写温度、控制读温度、通信改阈值时，若两条线同时碰到同一字段，可能读到「写到一半」的值，或丢掉某次更新。用 互斥锁（mutex：同一时刻只许一条线程进入临界区）包住读写：

pthread_mutex_lock(&st->lock);
temp = st->last_temp;
high = st->t_high;
pthread_mutex_unlock(&st->lock);

干净停：注册 SIGINT（Ctrl+C）处理函数，把 running=0；各线程循环判断该标志后退出；主线程 join 完毕后关风扇、释放 GPIO、断开 MQTT。验收时必须看到风扇停转再进程结束。

「为什么必须加锁、怎么亲眼看见竞态」，完整因果链放在下一节「深入理解」。

本节成果：知道共享字段进锁 · 能口述干净停步骤 · 带着「无锁会怎样」进入深入理解。
深入理解

竞态：板上必现错乱，电脑上用 TSan 指出两行

本节把三个词说清楚：竞态（data race）是两条线程无同步地并发读写同一共享变量，结果不确定甚至写乱；互斥锁（mutex）保证同一时刻只有一条线程进入临界区；ThreadSanitizer（TSan）是编译加上 -fsanitize=thread 的检查器，能在运行时报告「哪两行在抢同一变量」。板上交叉工具链未必开 TSan，所以产品现象在荔枝派 4A 上看，机理用主机上的 race-demo 演示。

三条证据链（必须都做）： (1) 板上无锁 → 日志/行为错乱（现象） (2) 板上加锁 → 行为正确（对照） (3) 电脑 TSan → 报告「这两行在抢同一变量」（机理）

图说明：只背「要加锁」不够；要先看见坏的，再修好，再用工具指出坏在哪两行。

(1)(2) 荔枝派 4A：无锁错乱 → 加锁正确

原因：采集线程写 last_temp，控制线程读 last_temp 并写 fan_on，通信线程可能同时改 t_high。没有锁时，这些访问可以交错发生。

机制（无锁时）：例如控制线程读到「写了一半」的浮点位型，温度跳成离谱值；或风扇状态与刚打印的温度互相矛盾。错乱可能间歇出现——「偶尔正常」不能证明没有竞态。

机制（加锁后）：所有对共享字段的读改都包在 pthread_mutex_lock / unlock 之间。同一时刻只有一条线程在临界区里，交错消失，阈值控温与 MQTT 改阈值恢复一致。

  1. 1
    板上先关锁
    按实验说明设 USE_LOCK 0（或等价开关），跑一段时间，记录温度跳变、风扇状态与温度矛盾等错乱现象。
  2. 2
    再开锁对照
    改为 USE_LOCK 1，所有共享读写进锁；错乱消失，风扇按阈值动作，MQTT 下行/上行仍正常。
无锁时可能交错（示意）： 采集: 写 temp 高位 ──┐ 控制: ├── 读到半新半旧 → 决策离谱 采集: 写 temp 低位 ──┘ 加锁后： 采集: lock → 写完整 temp → unlock 控制: lock → 读完整 temp → unlock （中间不会插入另一条线的半截写）

图说明：锁保护的是「一整次读改」的原子性观感，不是让三线变回单线程——三线仍并行，只是碰共享状态时排队。

(3) 电脑：race-demo + ThreadSanitizer 指出两行

chapters/ch06/code/race-demo/ 是故意让两条线程无锁抢同一计数器的小程序（无板、无 GPIO）。先无锁跑一遍，计数会偏小（丢更新）；再用 TSan 编译运行，报告会打印发生数据竞争的两处源码行。

$ cd chapters/ch06/code/race-demo
$ make
$ ./race-demo
# 无锁：最终计数明显小于期望（丢更新）

$ sudo sysctl vm.mmap_rnd_bits=28
# 固定步骤：新内核地址随机化位数与 TSan 冲突，调低后 TSan 才能稳定跑通

$ make tsan
$ ./race-demo
# ThreadSanitizer 打印数据竞争报告，标出两处访问行号

保存终端中 TSan 指出的两行，对照源码：那就是「谁和谁在无同步地碰同一变量」。产品路径（tri-thread）验收以加锁版本为准；不要把 race-demo 的无锁写法抄进板上骨架。

示意：TSan 报告在说什么 WARNING: ThreadSanitizer: data race Write of size 4 at 0x... by thread T1: #0 incr race-demo.c:42 ← 这一行在写 Previous read of size 4 at 0x... by thread T2: #0 incr race-demo.c:42 ← 同一变量，另一线程也在碰 （或另一行号：读/写交错）

图说明：TSan 不替你「修」程序，它只把竞态从「感觉偶尔不对」变成「具体是这两行」。你回去加 mutex，再跑应不再报同一竞争。

验收话术：「板上无锁时我看到过错乱；加锁后行为正确；电脑上 race-demo 经 TSan 指出抢同一变量的两行。所以共享状态必须进互斥锁，不是口头约定。」
race-demo 故意无锁，只用于教学演示。产品路径（tri-thread）以加锁版本验收。跑 TSan 前执行 sudo sysctl vm.mmap_rnd_bits=28 是本课固定步骤。
本节成果：能复述无锁→加锁→TSan 三步证据链 · 能解释 mutex、竞态与 mmap_rnd_bits 一步 · 带着验收清单进入本章实验。
