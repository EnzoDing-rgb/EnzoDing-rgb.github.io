3.1

外设与 I/O：从物理世界到程序

学习目标

I/O 模型

把「物理量 → 电气信号 → 软件可读数据」说清楚

MMIO 心智

知道设备常被映射成地址；CPU 用访存方式与之交互

用户态边界

Linux 下由内核驱动管理寄存器；应用走设备节点 / API

为后文铺路

理解后，3.2 的 gpiod 与 3.3 的 UART 不再像「魔法函数」

I/O：物理世界怎么进程序

嵌入式程序要和环境打交道：灯亮不亮、键有没有按下、温度多少、风扇转不转。这些都是 I/O——输入把外界变成程序里的数，输出把程序里的决定变成外界动作。

物理世界 电气层 软件层 温度 / 按键 → 电压 / 电平 → 驱动读寄存器或缓冲 LED / 风扇 ← 高/低、开关 ← 程序写「开/关」

同一条 GPIO 脚，配置成输出就能驱动 LED；配置成输入就能读按键。UART 则按字节流收发。传感器、执行器再复杂，对用户态程序来说，最终仍是「读到一个值」或「写出一个命令」。

MMIO 心智模型

MMIO（Memory-Mapped I/O，内存映射 I/O）把设备寄存器映射到 CPU 可见的地址空间。在 RISC-V SoC 上这很常见：驱动里对某个地址的读写，实际落到外设控制逻辑，而不是普通 RAM。

说法含义本课怎么用
MMIO用「访存」方式访问设备寄存器建立心智；用户态不直接碰这些地址
设备节点如 /dev/gpiochip0、/dev/ttyS1应用通过 open / ioctl / 库 API 访问
内核驱动独占硬件、做权限与时序管理保证多进程安全、屏蔽芯片细节
应用 C 代码 │ open / gpiod_* / read / write ▼ 设备节点（/dev/...） │ 系统调用 ▼ 内核驱动 ──► MMIO 寄存器 ──► 物理引脚 / 外设
记住一句：板上有 MMIO，但本课用户态程序不直接 *(volatile uint32_t*)addr = ...。你调用的是内核暴露的接口；寄存器细节由驱动完成。

用户态怎么碰硬件

在运行 Linux 的开发板上，权限与共享由内核统一管理。本课路径是：

  1. 确认设备节点存在（如 ls /dev/gpiochip*）。
  2. 用稳定 API 访问——GPIO 用 libgpiod，串口用 termios + 读写。
  3. 引脚编号以课程接线表为准，写进宏，方便换板。
安全边界：GPIO 多为 3.3 V 逻辑。不要把引脚直接接到市电或大电流负载；风扇等执行器必须经课程提供的驱动模块，GPIO 只控信号端。

GPIO 与 gpiod

GPIO 是可由软件配置为输入或输出的通用引脚。现代 Linux 推荐通过 libgpiod 访问字符设备 /dev/gpiochipN，而不是已弃用的 /sys/class/gpio。交叉编译时链接 -lgpiod；板端需有对应运行库。

应用（C） │ gpiod_chip_open / gpiod_line_request_output │ gpiod_line_set_value ▼ libgpiod ──► /dev/gpiochip0 │ ▼ 内核 GPIO 控制器 ──► 物理引脚 ──► LED

示例工程把关键配置收成宏，改接线只改这几行：

#define GPIO_CHIP_PATH  "/dev/gpiochip0"
#define LED_LINE        17   /* 以课程接线表为准 */

程序流程与 chapters/ch03/code/blink/main.c 一致：打开 chip → 取 line → 申请为输出 → 循环翻转电平 → 释放。默认闪约 20 次后退出；板端可用 Ctrl+C 提前结束（若你改成无限循环）。

  2. 2

    编辑 chapters/ch03/code/blink/main.c 中的 GPIO_CHIP_PATH、LED_LINE。
  3. 3

    见下方命令。预期：LED 周期性闪烁，终端打印 [INFO] blinking line …。

UART 与调试日志

UART 按约定波特率逐字节收发。第一章用 USB 转串口登录系统，那是控制台。本节示例打开板端某个 /dev/tty*，或先用 stdin/stdout 练同一套读写逻辑。

PC 串口工具 / 另一端 │ TX/RX 交叉 + GND ▼ /dev/ttyS?（板端） │ open + termios ▼ 你的 C 程序（Echo / 打日志） 更常见：printf → SSH 终端（stdout）

chapters/ch03/code/uart-echo/ 默认 USE_STDIO=1：在终端打字即可回显，适合没有第二路串口时练逻辑。改为 0 后打开 UART_DEV（默认 /dev/ttyS1），波特率 B115200。

$ cd chapters/ch03/code/uart-echo
$ make clean && make
$ scp uart-echo user@board-ip:~/
$ ssh user@board-ip './uart-echo'

日志习惯：关键行带前缀，例如 [INFO]、[ERR]。温度、阈值、风扇状态各打清楚，实验里对照现象会轻松很多。

接线：真串口时 TX↔RX 交叉，GND 共地；两端波特率一致；电平多为 3.3 V，勿接 RS-232 高压口。不要占用正在当系统控制台的那一路，除非课程明确要求。

传感与执行：概念与数据流

DHT22 用单总线报温度、湿度。用户态位带可能偶发失败——属预期。读失败时应打 [ERR] 并进入下一采样周期，让程序继续跑。

执行侧本课主线是开/关：GPIO 高/低 → 继电器或开关模块 → 风扇电源通断。PWM 用占空比调平均功率，讲义只建立概念；综合项目可再扩展调速。

开/关GPIO → 驱动模块接通或断开风扇本章实验主线：实现简单、验收清晰
PWM快速开关，占空比改变平均电压讲义建立概念；后续可扩展
DHT22 ──► 温度 T、湿度 H │ ▼ 阈值判断（概念） T > T_HIGH → 风扇开 T < T_LOW → 风扇关 │ ▼ GPIO + 驱动模块 ──► 风扇 讲义：理清数据流，可打印一次温度 实验：周期采样 + 主循环闭环
/* 伪代码：闭环逻辑在 lab 的 temp-fan 里落地 */
if (读温度失败) { 打 [ERR]; 继续下一轮; }
else if (T > T_HIGH) 风扇_开();
else if (T < T_LOW)  风扇_关();
/* T_LOW ~ T_HIGH 之间保持上一状态，避免临界抖动 */
边界：本节把 UART 日志习惯、传感与执行的角色和数据流讲清楚即可。不要在讲义里拼完整温控主循环——接线检查、周期采样、阈值开/关与验收见 lab.html 与 code/temp-fan/。阈值目前可写在宏里；第四章再外置配置。
