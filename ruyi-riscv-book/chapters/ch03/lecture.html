3.1

GPIO 输入输出：LED 与按键

学习目标

GPIO 角色

区分输出（驱动 LED）与输入（读按键）

libgpiod

用字符设备接口申请、读写引脚（优于旧 sysfs）

消抖直觉

机械按键抖动时如何简单过滤

可移植配置

chip 名与 line 号用宏/参数，不写死「某板某脚」

GPIO 与 gpiod

GPIO（General-Purpose Input/Output）是可由软件配置为输入或输出的芯片引脚。输出可驱动 LED、继电器；输入可读按键、部分传感器电平。

在现代 Linux 上，推荐通过 libgpiod（字符设备 /dev/gpiochipN）访问 GPIO，而不是已弃用的 /sys/class/gpio。板端需安装开发包（如 libgpiod-dev）与运行库；交叉编译时链接 -lgpiod。

应用（C） │ gpiod_chip_open / gpiod_line_request_* ▼ libgpiod ──► /dev/gpiochip0（或 chip1…） │ ▼ 内核 GPIO 控制器 ──► 物理引脚 ──► LED / 按键

引脚怎么定？

不同板卡的「丝印脚号」与 gpiochip 的 line 编号不是同一套数字。本课示例用宏占位：

#define GPIO_CHIP_PATH  "/dev/gpiochip0"
#define LED_LINE        17   /* 以课程接线表 / 板卡文档为准 */
#define BTN_LINE        27

LicheePi 4A 等板的具体对应关系，以课程当堂接线表或厂商外设文档为准；改宏后重新交叉编译即可，不必改逻辑。

安全：先确认电压（多为 3.3 V 逻辑）。不要把 GPIO 直接接到市电或大电流负载；风扇等执行器必须经课程提供的驱动模块（继电器/MOSFET），由 GPIO 只控「信号端」。

操作步骤

  1. 1
    确认板端有 gpiochip
    ssh user@board-ip 'ls /dev/gpiochip*'；可选安装 gpiodetect / gpioinfo 查看 line 名称。
  2. 2
    按接线表改宏
    编辑 chapters/ch03/code/blink/main.c 中的 GPIO_CHIP_PATH、LED_LINE（及按键示例中的 BTN_LINE）。
  3. 3
    交叉编译并部署
    cd chapters/ch03/code/blink && make clean && make，再 scp 到板端运行。预期：LED 周期性闪烁。
  4. 4
    按键消抖（课堂扩展）
    读输入时，连续两次采样间隔约 20–50 ms 且电平一致再认定「按下」，避免抖动误触发。
# 主机
$ cd chapters/ch03/code/blink
$ make clean && make
$ file blink
$ scp blink user@board-ip:~/
$ ssh user@board-ip './blink'
权限：若提示无法打开 /dev/gpiochip*，将用户加入 gpio 组，或以课程说明的方式授权；课堂可用临时方案，但勿养成长期 sudo 跑业务程序的习惯。

课堂练习

  1. 为何讲义用 LED_LINE 宏，而不是在代码里写死「LicheePi 4A 的某某脚」？
  2. 按键消抖：若采样间隔过短，可能出现什么现象？
本节成果：能说明 GPIO 输入/输出差异 · 用 blink 工程点亮 LED · 知道引脚以接线表为准 · 了解简单消抖思路。
3.2

UART 与串口调试日志

学习目标

UART 用途

区分「系统控制台串口」与「应用自开的串口设备」

termios

设置波特率、原始模式，完成 Echo

调试日志

用串口或标准输出打印带标签的运行信息

UART 角色

UART 按约定波特率逐字节收发。第一章用 USB 转串口登录系统，那是控制台。本章示例打开的是板端某个 /dev/tty*（如 ttyS1、ttyUSB0），做应用层收发——具体设备名以板卡文档与课程接线为准。

PC 串口工具 / 另一端设备 │ TX/RX 交叉 + GND ▼ /dev/ttyS?（板端） │ open + termios ▼ 你的 C 程序（Echo / 打日志）

实验与综合项目里，更常见的是：程序把日志打到 stdout（SSH 终端可见），或同时写文件（第四章）。串口 Echo 训练的是「会配 termios、会读写设备节点」——排错时同样有用。

操作步骤

  1. 1
    确认设备节点
    板端 ls /dev/ttyS* /dev/ttyUSB*；不要占用正在当系统控制台的那一路，除非课程明确要求。
  2. 2
    编译 uart-echo
    cd chapters/ch03/code/uart-echo && make。源码里 UART_DEV、BAUD（默认 115200）按实际修改。
  3. 3
    板端运行并回显
    一端跑程序，另一端用串口工具发字符，应原样回显。无第二路串口时，可先把程序改为「读 stdin、写 stdout」验证逻辑，再换真串口。
  4. 4
    日志习惯
    关键行带前缀，例如 [INFO]、[ERR]；温度、阈值、风扇状态各打一行，方便实验 1 对照。
$ cd chapters/ch03/code/uart-echo
$ make clean && make
$ scp uart-echo user@board-ip:~/
$ ssh user@board-ip './uart-echo'   # 或按课程说明接好第二路串口再跑
接线：TX↔RX 交叉，GND 共地。波特率两端必须一致。电平多为 3.3 V，勿接 RS-232 高压口。

课堂练习

  1. SSH 已经能看 printf 输出时，为什么还要学配置 /dev/tty*？
  2. Echo 正常但乱码，优先检查哪两项？
本节成果：能打开并配置一路串口 · 完成 Echo 或等价验证 · 养成带标签的调试打印习惯。
3.3

传感与执行：DHT22 与 PWM / 开关输出

学习目标

DHT22

理解单总线读温湿度的时序约束与失败重试

PWM 概念

占空比如何改变平均功率（调光/调速）

开/关控制

用 GPIO 驱动继电器/模块实现风扇启停（实验 1 主线）

阈值逻辑

温度高于上限开、低于下限关；组装见 lab

DHT22 温湿度

DHT22（AM2302）用一根数据线与主机通信，对时序要求严。用户态用 GPIO 位bang 时，可能受调度影响导致偶发读失败——正常现象；应重试并打 [ERR] 日志，而不是静默忽略。

讲义阶段：能调用课程提供的读取函数（或 temp-fan 中的封装）打印一次温度、湿度即可。周期采样、阈值控风扇放在 lab。

DHT22 ──单总线──► GPIO（输入/输出切换） │ ▼ 温度 T、湿度 H（浮点或定点） │ ▼ 讲义：打印一次 实验：循环采样 + 阈值判断

PWM 与开/关：两种「控风扇」

方式做法本课用法
开/关GPIO 高/低 → 驱动模块接通或断开风扇电源实验 1 主线：实现简单、验收清晰
PWM快速开关，用占空比调平均电压 → 转速/亮度变化讲义建立概念；综合项目可再扩展调速

风扇与驱动模块由课程套件提供。GPIO 只接到模块的信号端；电源按套件说明接，勿用 GPIO 直接灌大电流。

Linux 上 PWM 常通过 /sys/class/pwm/ 导出通道；若板卡未启用对应 overlay，课堂以开/关为主即可，不必强行调通 PWM。

阈值控制（逻辑，不在此节写完整程序）

/* 伪代码：实验 1 主循环核心 */
if (读温度失败) { 打日志; 继续下一轮; }
else if (T > T_HIGH) 风扇_开();
else if (T < T_LOW)  风扇_关();
/* T_LOW ~ T_HIGH 之间保持上一状态，避免临界抖动 */

完整接线检查、周期、异常处理与验收表 → lab.html + code/temp-fan/。

课堂练习

  1. 为何实验 1 主线选「开/关」而不是一上来就 PWM 调速？
  2. DHT22 偶发读失败时，程序应如何表现才算「可诊断」？
