5.1

Broker、主题、发布订阅

发布订阅模型

MQTT 不强调「你直接连我」，而强调「大家都连中转站」。Broker 是消息中转站（本课常用软件 Mosquitto）：客户端连上它，按主题（频道名，如 course/led/cmd）转发消息。publish（发布）是往某个主题发一条消息；订阅是盯着某个主题收消息。

手机 App（发布者） │ publish course/led/cmd payload="on" ▼ Broker（Mosquitto） │ 转发给所有订阅者 ▼ 荔枝派 4A 程序（订阅者）→ GPIO → 外接 LED 反方向： 板子 publish course/led/status payload="on" → Broker → 手机 / 桌面客户端看见状态
术语一句话本课例子
Broker消息中转站主机或板上的 Mosquitto，默认端口 1883
主题频道名course/led/cmd、course/led/status
publish（发布）往某主题发一条手机发 on；板子报状态
订阅盯着某主题收板子订 course/led/cmd
payload消息正文on / off

启动 Mosquitto（主机示例）

# Debian/Ubuntu 示例；以你发行版包名为准
$ sudo apt update
$ sudo apt install -y mosquitto mosquitto-clients
$ sudo systemctl enable --now mosquitto

# 本机自测：一终端订阅，另一终端发布
$ mosquitto_sub -h 127.0.0.1 -t 'course/led/cmd' -v
$ mosquitto_pub -h 127.0.0.1 -t 'course/led/cmd' -m 'on'
网络：板子连接 Broker 时用主机局域网 IP，不要写 127.0.0.1（那是板子自己）。防火墙需放行 1883。课堂若提供统一 Broker 地址，以当堂板书为准。
本节成果：能画发布—Broker—订阅三角 · Mosquitto 能收发一条测试消息 · 记下 Broker IP 与主题名。

下行：订阅命令驱动外接 LED

主题与 payload 约定

本课默认：

  • 命令主题：course/led/cmd
  • payload：on → 灯亮；off → 灯灭
  • 未知 payload → 打 [ERR]，不改变当前电平
mosquitto_pub / 手机 App │ topic=course/led/cmd msg=on ▼ Broker ▼ mqtt-led 收到消息 │ 比较 payload 是否为 on / off ▼ gpiod_line_set_value(LED, 1 或 0)

5.3

上行：发布状态

灯态变化后 publish

灯亮灭改变后，板子向 course/led/status publish（发布）当前状态（如 on/off）。订阅该主题的客户端即可确认「命令是否落地」。脚手架中 publish_status() 为桩，实验里接到点灯成功之后调用。

/* 伪代码 */
led_set(1);
publish_status("on");   /* TODO: mosquitto_publish(...) */
# 主机订阅状态
$ mosquitto_sub -h BROKER_IP -t 'course/led/status' -v

应用层「发布成功」只说明库函数返回了；它在网上究竟怎么走，要靠下一节用抓包把分层拆开看。

本节成果：状态主题可被订阅端看见 · 知道上行与下行用不同主题 · 带着「一次 publish 在网上长什么样」进入深入理解。
深入理解

一次 publish：从应用到以太网，再用 tcpdump 看见

先把本节会反复用到的词说清楚：Broker 是 MQTT 中转站；主题 是频道名；publish 是往某主题发一条消息；tcpdump 是在网卡上抓原始包、打印协议字段的工具。默认明文 MQTT 跑在 TCP 端口 1883 上。

原因 → 机制：一次 publish 怎么一层层包下去

你在板子上调用 mosquitto_publish(...) 时，应用只交出「主题 + payload」。库和内核再把它装进 MQTT 报文，塞进 TCP 连接（目的端口 1883），外面再套 IP，最外层是以太网帧（有线）或 Wi-Fi 帧。Broker 拆包后按主题转发给订阅者。

应用：mosquitto_publish(topic="course/led/status", payload="on") │ ▼ MQTT 报文（应用层：主题名、payload、报文类型等） ▼ TCP 段（传输层：源端口、目的端口 1883、序号、载荷） ▼ IP 包（网络层：源 IP、目的 IP） ▼ 以太网帧（链路层：MAC、类型、帧载荷） │ 网线 / Wi-Fi ──► Broker 主机

图说明：每一层只多加自己的头尾，并不改写你业务上的「on」含义；tcpdump 能同时看到这些头，所以能对照「应用里的一次 publish」和「网上的一串包」。

验证：标注一份看起来像 tcpdump 的输出

在 Broker 主机或板端（以能看到板→Broker 流量为准）开抓包，再触发一次上行 publish：

$ sudo tcpdump -i any -nn -A port 1883 -c 20

下面是一份示意输出（字段形状与真实 tcpdump 相近；IP/时间以你机器为准）。右侧注释标出各层：

14:02:11.123456 IP 192.168.1.50.45678 > 192.168.1.10.1883: Flags [P.], ...
                 │                │              │     │
                 │                │              │     └─ 目的端口 1883 → TCP 上的 MQTT
                 │                │              └─ Broker 主机 IP（网络层目的）
                 │                └─ 板子临时源端口（传输层）
                 └─ 板子 IP（网络层源）

    …… TCP 载荷的一部分（-A 以 ASCII 显示）……
    ....course/led/status..on....
         │                 │
         │                 └─ payload 痕迹（明文 MQTT 时可能露出）
         └─ 主题名痕迹（应用层 MQTT）

逐项对照：

你在输出里看到的对应哪一层说明什么
IP a.b.c.d > e.f.g.h网络层（IP）谁发给谁：板子 → Broker
.1883（目的端口）传输层（TCP）这是默认 MQTT 端口，不是随便一个网页流量
Flags [P.] 等TCP段在推送数据；握手/ACK 包也可能出现在同一段抓取里
ASCII 里的主题 / on应用层（MQTT）业务内容落在 TCP 载荷里；明文时可见片段
（有线抓包时的 MAC 行，若开了相应选项）链路层（以太网）帧在本地网段上传递；本课用 -nn -A port 1883 已够指认 TCP/IP/MQTT
  1. 1
    开抓包
    sudo tcpdump -i any -nn -A port 1883 -c 20
  2. 2
    触发一次上行
    让板子 publish 状态（或主机 mosquitto_pub 发一条到状态主题）。
  3. 3
    对照上表
    指出 1883、双方 IP，以及 ASCII 区是否露出主题或 payload；用自己的话复述「MQTT → TCP → IP → 出网卡」。
验收话术：「这次 publish 的内容在 MQTT 应用层；它装在 TCP（1883）里，再装在 IP 里，最后由以太网/Wi-Fi 发出。tcpdump 让我看见端口与载荷痕迹，而不是只相信库函数返回成功。」
抓包需相应权限；只抓本课端口、限制 -c 包数。不要在公共网络长时间无过滤抓包。若走 TLS（常 8883），ASCII 一般看不到明文 payload——那是加密层在起作用，分层关系不变。
本节成果：能画出 publish 的分层包装 · 能标注一份 tcpdump 片段里各字段属于哪一层 · 保存一段含 1883 的输出进入本章实验验收。
