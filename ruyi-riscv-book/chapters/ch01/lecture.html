平台分层：从 ISA 到应用

RISC-V 是开放指令集架构（ISA），定义了处理器执行指令的接口。芯片和开发板是不同层次的概念。建立分层思维是后续排查「镜像与板卡不匹配」「工具链架构错误」等问题的基础。

RISC-V ISA（开放指令集架构规范） ↓ 处理器核 / SoC（TH1520） ↓ 开发板（LicheePi 4A） ↓ 操作系统（RevyOS） ↓ 开发工具（ruyi、编译器、SSH、串口工具）

ruyi：包管理与环境管理入口

ruyi 是 RuyiSDK 的包管理入口，负责发现和获取 RISC-V 开发所需的软件包、镜像与工具链；真正编译代码的是 GCC。两者关系类似「应用商店」与「应用程序」。ruyi 之下还有一层自动化能力——一键准备（写镜像）与一键编译（便捷构建），1.2 与 1.3 会分别引入。

ruyi 命令 ├─ 软件源与包信息 → 查询可用工具链、镜像与设备支持 ├─ 工具链获取 → 下载并安装 RISC-V GNU Toolchain ├─ 环境与配置管理 → 管理不同项目的工具链环境 └─ 设备支持查询 → 查看受支持的开发板与一键准备方式 ↓ GCC / binutils / sysroot

项目要求检查方式
主机环境Linux x86_64（推荐），macOS / Windows 备选uname -a

网络可访问 RuyiSDK 软件源浏览器或连通性测试

常见陷阱
· 部分 USB-C 线仅供电、无数据通道 → 无法识别设备时先换线
· 串口电平必须为 3.3V → 5V 可能损坏板卡
· 工具链前缀应选 linux-gnu（Linux 应用）→ unknown-elf 是裸机工具链，不适用

操作步骤

  1. 1
    记录主机环境
    uname -a / uname -m — 确认主机架构，记入个人环境清单。
  2. 2
    安装 ruyi 并确认命令位置
    command -v ruyi 确认 shell 实际调用的文件。ruyi --version 记录版本号。
  3. 3
    更新软件源并查询工具链
    ruyi update 更新源。ruyi list | grep toolchain 搜索可用工具链。
  4. 4
    安装并验证编译器
    riscv64-unknown-linux-gnu-gcc -dumpmachine 应返回 riscv64-unknown-linux-gnu，这就是本课统一使用的目标三元组。
  5. 5
    最小编译验证
    编译 int main(void) { return 0; } → file 检查产物为 RISC-V ELF。在主机上 ./a.out 会因架构不同而无法运行——用 file 验收即可。
  6. 6
    平台分层图与接口标注
    按上文分层图绘制个人版本，在 LicheePi 4A 资料上标出电源、串口、网络与启动介质。
# 主机架构
$ uname -a

# ruyi 安装位置与版本
$ command -v ruyi
$ ruyi --version

# 更新源并查询工具链
$ ruyi update
$ ruyi list | grep toolchain

# 工具链三元组
$ riscv64-unknown-linux-gnu-gcc -dumpmachine
riscv64-unknown-linux-gnu

# 最小编译验证
$ printf 'int main(void) { return 0; }\n' > mini.c
$ riscv64-unknown-linux-gnu-gcc mini.c -o mini-riscv
$ file mini-riscv
mini-riscv: ELF 64-bit LSB executable, UCB RISC-V, ...

镜像烧录：分区级数据写入

烧录工具将引导程序、内核、设备树和根文件系统按分区结构写入目标介质。建议的烧录流程：校验镜像 → 识别目标设备 → 严格按发布说明执行 → 确认工具有成功报告。

镜像发布包 ├─ 引导程序 ← 上电后第一条指令 ├─ Linux 内核与设备树 ← 硬件初始化 + 驱动匹配 └─ RevyOS 根文件系统 ← 用户空间全部内容 ↓ 烧录（按分区写入） eMMC / SD / 其他启动介质 ↓ 上电启动 串口日志 → 登录界面

三条通道，三个角色

串口（UART）

本地调试 · 最后保障线

不依赖网络，直接观察启动日志。接线：GND-GND、TX→RX、RX→TX，3.3V 电平。

SSH

远程主工作通道

加密网络终端，适合日常开发。Ed25519 密钥登录替代密码。首次连接须核对主机指纹。

SCP

文件传输通道

复用 SSH 连接传输文件。上传后板端 sha256sum 校验——两端哈希一致即传输无损坏。

SSH 密钥登录配置

# 板端启用 SSH 服务
$ sudo systemctl enable --now ssh

# 主机生成 Ed25519 密钥对
$ ssh-keygen -t ed25519 -C "riscv-course"

# 首次连接——核对主机指纹，确认无误后再输入 yes
$ ssh user@board-ip

# 部署公钥
$ ssh-copy-id user@board-ip

# 免密验证
$ ssh user@board-ip 'hostname; uname -m'
指纹变更警告：可能是开发板重装系统，也可能连错了设备。先核对 IP 和串口中的主机指纹，再清理 known_hosts 旧记录。

SCP 双向传输校验

# 上传
$ printf 'transfer test\n' > test.txt
$ sha256sum test.txt
$ scp test.txt user@board-ip:/tmp/
$ ssh user@board-ip 'sha256sum /tmp/test.txt'   # 哈希应一致

# 下载
$ scp user@board-ip:/etc/os-release ./board-os-release.txt

交叉编译工作流

交叉编译是「在主机上编译，在目标板上运行」：主机架构（x86_64）与目标架构（riscv64）不同，因此编译器、汇编器、链接器都必须使用面向 RISC-V 的版本——这正是 1.1 中安装的 riscv64-unknown-linux-gnu-gcc 系列工具。

源代码（.c） ↓ riscv64-unknown-linux-gnu-gcc（主机上运行） RISC-V 可执行文件（ELF） ↓ scp 传输 LicheePi 4A（riscv64，实际运行）

典型流程：编写/获取源码 → 主机交叉编译 → file 确认产物架构 → SCP 传输到板端 → SSH 登录运行 → 记录结果。这条链路与 1.2 建立的三通道完全对应。

Makefile 模板：固定交叉编译参数

把交叉编译命令写进 Makefile，避免每次手敲冗长参数，也便于团队统一工具链版本。

# Makefile — 交叉编译模板
CROSS_COMPILE := riscv64-unknown-linux-gnu-
CC            := $(CROSS_COMPILE)gcc
CFLAGS        := -O2 -Wall
TARGET        := app

$(TARGET): main.c
	$(CC) $(CFLAGS) main.c -o $(TARGET)

clean:
	rm -f $(TARGET)

.PHONY: clean
# 使用
$ make
$ file app
app: ELF 64-bit LSB executable, UCB RISC-V, ...
常见陷阱：忘记切换 CC 会默默用主机 GCC 编译，产物在板端「文件损坏」或直接拒绝执行——先用 file 检查架构，再排查运行问题。

CoreMark：嵌入式算力基准

CoreMark 是业界常用的嵌入式处理器算力基准测试，输出一个可比较的分数（iterations/sec），用于衡量不同芯片、不同优化选项下的处理器性能。本课用它作为交叉编译 + 上板运行的综合练习：分数本身不是重点，完整走通「编译 → 传输 → 运行 → 读数」链路才是。

构建 CoreMark for riscv64

CoreMark 源码不在本课程仓库中，需自行获取（官方仓库或课程指定镜像地址）。以下是典型构建路径：

# 获取源码（以官方仓库为例，实际地址以课程指引为准）
$ git clone https://github.com/eembc/coremark.git
$ cd coremark

# 交叉编译：通过 make 变量指定工具链前缀
$ make PORT_DIR=linux64 \
       CC=riscv64-unknown-linux-gnu-gcc \
       LD=riscv64-unknown-linux-gnu-gcc \
       CFLAGS="-O2 -DPERFORMANCE_RUN=1" \
       XCFLAGS="-DFLAGS_STR=\"cross-riscv64\""

# 确认产物架构
$ file coremark.exe
coremark.exe: ELF 64-bit LSB executable, UCB RISC-V, ...

产物文件名因 CoreMark 版本而异（常见为 coremark.exe 或 coremark_PORT_DIR.exe），以实际生成结果为准。构建成功后，用 1.2 学到的 SCP 把可执行文件传到板端，SSH 登录运行并记录分数——完整验收步骤见 lab.html。
