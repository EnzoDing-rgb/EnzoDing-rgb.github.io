2.1

类型、变量、控制流

类型与变量

程序处理的是有类型的数据。本课最常见的几种：

类型典型用途例子
int计数、状态、返回码风扇开关 0/1、循环下标
float / double温度、阈值28.5 °C
char / 字符串日志文字"FAN ON"
int 返回值成功 / 失败0 成功，非 0 失败
int fan_on = 0;          /* 0 = 关，1 = 开 */
float temp_c = 27.5f;    /* 当前温度 */
const float T_HIGH = 28.0f;
const float T_LOW  = 26.0f;
习惯：配置常量用 const 或宏写清楚含义；布尔语义用 0/1 并在注释里标明「开/关」。

控制流：判断与循环

温控的核心就是比较与分支：

if (temp_c > T_HIGH) {
    fan_on = 1;
} else if (temp_c < T_LOW) {
    fan_on = 0;
}
/* 落在 [T_LOW, T_HIGH] 之间：保持现状（滞回，避免频繁抖动） */

周期性采样用循环：

for (int i = 0; i < n; i++) {
    /* 读一次温度，做一次决策，打印一行 */
}

while (running) {
    /* 主循环：采样 → 决策 → 延时 */
}
温度序列示例（滞回） T_HIGH = 28 T_LOW = 26 25 → 关 27 → 保持 29 → 开 27 → 保持 25 → 关 关键：比较方向写反、用错阈值字段，都会让「该开不开 / 该关不关」。

小练习

  1. 写出：温度严格大于 28 开风扇，严格小于 26 关风扇，中间保持原状态。
  2. temp >= T_HIGH 与 temp > T_HIGH 在边界 28.0 上行为差在哪里？
  3. 若把 T_HIGH 和 T_LOW 写反，会出现什么现象？
本节成果：能读懂阈值判断与简单循环；知道滞回区间要「保持现状」。
2.2

函数、头文件与多文件工程

函数：把「一件事」包起来

函数有参数、返回值和函数体。本课常见写法：

/* 返回 0 成功，非 0 失败 */
int read_temp(float *out_c);

void apply_fan_policy(float temp_c, int *fan_on);

调用方根据返回值决定是否继续用读到的温度：

float t;
if (read_temp(&t) != 0) {
    /* 读失败：打日志，不要用未初始化的 t */
    return;
}
apply_fan_policy(t, &fan_on);

头文件：声明接口

头文件告诉其他 .c：「这些函数/类型存在」。实现放在对应的 .c 里。用 include guard 防止重复包含：

/* greet.h */
#ifndef GREET_H
#define GREET_H

void greet(const char *name);

#endif

声明在 .h

函数原型、结构体类型、对外宏——给调用方看。

定义在 .c

函数体、静态内部变量——实现细节。

多文件工程：跟做模板

跟做目录：

  • chapters/ch02/code/hello/ — 单文件入门
  • chapters/ch02/code/project-template/ — src/ + include/ + build/
project-template/ ├── Makefile ├── include/greet.h ← 接口声明 ├── src/main.c ← 调用 greet() ├── src/greet.c ← 实现 greet() └── build/ ← make 生成，可删
$ cd chapters/ch02/code/hello
$ make CROSS_COMPILE=     # 本机 gcc；省略则用交叉前缀

2.3

指针、数组、结构体与本课代码怎么读

指针与数组

指针是「存着某个对象地址」的变量。函数要写回调用方的变量时，就传指针：

void set_fan(int *fan_on, int value)
{
    if (fan_on == NULL)
        return;
    *fan_on = value;   /* 改的是调用方那份变量 */
}

数组是连续同类型元素。假温度序列常用数组驱动：

static const float seq[] = { 25.0f, 27.0f, 29.0f, 27.0f, 24.0f };
for (size_t i = 0; i < sizeof(seq)/sizeof(seq[0]); i++) {
    /* 用 seq[i] 当本轮温度 */
}
解引用前先确认指针非 NULL；数组下标不要越界。闯关脚手架里故意埋了相关坑。

结构体：把相关字段捆在一起

配置不必散落成一堆全局变量：

typedef struct {
    float t_high;      /* 高于此开风扇 */
    float t_low;       /* 低于此关风扇 */
    int   period_ms;   /* 采样周期（毫秒） */
} ThermoConfig;

void decide(const ThermoConfig *cfg, float temp, int *fan_on)
{
    if (temp > cfg->t_high)
        *fan_on = 1;
    else if (temp < cfg->t_low)
        *fan_on = 0;
}

读结构体字段用 cfg->t_high（指针）或 cfg.t_high（变量本身）。用错字段名是真实 bug 的常见来源。

本课代码怎么读

拿到一份半成品，建议按这个顺序扫：

  1. 1
    看头文件 — 有哪些结构体、API、返回值约定。
  2. 2
    看 main — 输入从哪来、如何循环、打印什么。
  3. 3
    看实现 — 比较方向、字段名、空指针、错误分支是否完整。
  4. 4
    对照行为表 — 改一处、跑一次，输出是否逐步贴近验收表。
读代码清单（闯关时可用） [ ] 阈值比较是 > / < 还是写反了？ [ ] 用的是 t_high / t_low，还是误用了 period_ms？ [ ] 指针参数有没有判 NULL？ [ ] 传感器失败时有没有跳过决策？
下一步：打开 lab.html，进入 fake-thermostat/ 改代码闯关。软件输入、无真 DHT——专注把逻辑改对。

课堂练习

  1. float *p = NULL; *p = 1.0f; 会发生什么？应如何防护？
  2. 为何要把 t_high、t_low、period_ms 放进同一个结构体？
  3. 函数返回错误码后，调用方若仍使用未写入的输出参数，错在哪里？
本节成果：能读指针写回、数组序列与配置结构体；知道按「头文件 → main → 实现 → 行为表」排查半成品。
