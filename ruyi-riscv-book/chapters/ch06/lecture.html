6.1

一条线会互相堵

学习目标

单线程瓶颈

说清「等网络」如何拖死「读传感器」

对照第五章

mosquitto_loop_forever 占住主线程的含义

动机

为何要拆成多条执行流

边界

本章用线程，不引入多进程架构

阻塞从哪来

第五章的 mqtt-led 用 mosquitto_loop_forever 一直处理网络事件——这很好，但同一条主线若还要每 2 秒读 DHT、判阈值、开风扇，就会互相抢时间：网络回调慢了，采样推迟；采样位带耗时长了，MQTT 心跳可能超时。

单线程硬塞在一起： while (1) { mqtt_处理一会儿(); // 可能卡住 读_DHT22(); // 也可能卡住 控风扇(); } 结果：某一段一堵，后面全停。

线程让进程里同时跑多段逻辑：采集线专心采样，控制线专心判阈值，通信线专心 MQTT。它们通过共享内存里的状态结构交换数据——下一节定义这份结构，再下一节处理「同时改会写乱」的问题。

和第四章 select 的关系：select 适合「一个线程盯几个 I/O」；当业务块变多（传感时序 + 网络库自带循环），用线程按职责切开更清晰。两种手段都要会。

课堂练习

  1. 列出三种会让「单线程温控+MQTT」卡住的具体调用。
  2. 若只用 select、不用线程，能否覆盖 mosquitto 的网络循环？需要什么条件？
本节成果：能解释单线程互相堵的现象 · 明确本章用三线程拆分的动机。
6.2

三线：采集∥控制∥通信(MQTT)

学习目标

采集线程

周期读 DHT，写入共享温度

控制线程

读温度与阈值，驱动风扇

通信线程

跑 MQTT 循环：下行命令、上行状态

共享结构

温度、风扇、阈值、运行标志

职责拆分

┌─────────────┐ 写 last_temp ┌─────────────┐ │ 采集线程 │ ───────────────► │ │ │ 读 DHT22 │ │ 共享状态 │ └─────────────┘ │ + 互斥锁 │ │ │ ┌─────────────┐ 读温度/写风扇 │ │ │ 控制线程 │ ◄──────────────► │ │ │ 阈值开/关 │ │ │ └─────────────┘ │ │ │ │ ┌─────────────┐ 改阈值/发状态 │ │ │ 通信线程 │ ◄──────────────► │ │ │ MQTT loop │ └─────────────┘ └─────────────┘
线程做什么不做什么
采集按周期读传感器，更新 last_temp/last_hum不直接开风扇、不跑网络库
控制读温度与阈值，调用风扇 GPIO不阻塞在 MQTT
通信mosquitto_loop；下行改阈值或灯；上行发布状态不做 DHT 位带

骨架一览

chapters/ch06/code/tri-thread/ 给出 pthread 创建、共享结构、三线程函数空壳与信号退出标志。实验中补全各线程循环与加锁读写。

struct shared_state {
    pthread_mutex_t lock;   /* 锁：同一时刻只许一条线改关键字段 */
    float last_temp;
    float t_high, t_low;
    int fan_on;
    volatile int running;   /* Ctrl+C 置 0，各线程退出 */
};
$ cd chapters/ch06/code/tri-thread
$ make clean && make
$ file tri-thread
$ scp tri-thread user@board-ip:~/
线程函数必须是 void *(*)(void *)；创建后用 pthread_join 等其结束。风扇与 GPIO 初始化放在开线程之前，避免多线同时申请同一 line。

课堂练习

  1. 若采集线程直接调用 fan_set，破坏了哪条职责边界？
  2. 通信线程收到「提高上限」时，应改共享结构的哪个字段？谁去真正开风扇？
本节成果：能画出三线与共享状态 · 能编译部署 tri-thread 骨架 · 知道各线程边界。
6.3

共享状态与干净停 · 深入理解：竞态与 ThreadSanitizer

学习目标

竞态

两条线同时改同一数据，结果写乱

互斥锁

加锁后再读改共享字段

干净停

Ctrl+C 置标志 → 关风扇 → 线程退出

深入理解

板上必现错乱再修好；电脑 TSan 指出抢变量的行

锁与干净停

采集写温度、控制读温度时，若中间被打断，可能读到「写到一半」的值。用 互斥锁（mutex：同一时刻只许一条线进入临界区）包住读写：

pthread_mutex_lock(&st->lock);
temp = st->last_temp;
high = st->t_high;
pthread_mutex_unlock(&st->lock);

干净停：注册 SIGINT（Ctrl+C）处理函数，把 running=0；各线程循环判断该标志后退出；主线程 join 完毕后关风扇、释放 GPIO、断开 MQTT。验收时必须看到风扇停转再进程结束。

深入理解：板上不加锁 → 加锁；电脑上 TSan

本课要求你亲眼看见问题，而不是背「要加锁」三个字：

  1. 1
    板上（tri-thread）
    按实验说明先关掉锁（或走「故意无锁」编译开关），跑一段时间，记录温度/风扇日志出现跳变、矛盾或明显错乱。
  2. 2
    加锁修好
    恢复所有共享读写的 lock/unlock，错乱消失；风扇阈值行为恢复正常。
  3. 3
    电脑上跑 race-demo
    chapters/ch06/code/race-demo/ 是确定性强制抢同一计数器的小程序（无板、无 GPIO）。本机： 
    cd chapters/ch06/code/race-demo
make
./race-demo          # 无锁：计数偏小（丢更新）
sudo sysctl vm.mmap_rnd_bits=28   # 固定步骤：让 TSan 稳定跑通
make tsan
./race-demo          # ThreadSanitizer 打印数据竞争报告
    保存终端中 TSan 指出的两处访问行号，对照源码。
深入理解三条证据链： 板上无锁 → 日志/行为错乱（现象） 板上加锁 → 行为正确（对照） 电脑 TSan → 报告「这两行在抢同一变量」（机理）
ThreadSanitizer（TSan）：编译加上 -fsanitize=thread 的检查器。它在电脑上跑同一类抢数据程序，打印哪两行在并发访问同一变量。板上交叉工具链未必开 TSan，故竞态机理用主机演示。
race-demo 故意无锁；不要把无锁写法抄进产品骨架。产品路径（tri-thread）验收以加锁版本为准。

课堂练习

  1. 为何「偶尔看起来正常」不能证明没有竞态？
  2. 干净停时若只退出通信线程、不管控制线程，可能留下什么硬件状态？
本节成果：理解锁与 running 标志 · 能复述深入理解三步 · 带着验收清单进入本章实验。
